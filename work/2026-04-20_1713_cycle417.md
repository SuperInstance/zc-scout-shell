## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph**. However, the trend is shifting from monolithic orchestrators to **specialized, composable modules**. Newer projects focus on specific niches: memory management, tool execution, or low-latency inference, rather than attempting to be full-stack solutions.

### 2. Notable New Entrants & Trends
*   **HyperAgent** (GitHub: `hyperagent-org/hyperagent`): A Rust-based framework emphasizing deterministic execution and sub-millisecond latency for high-frequency agent tasks. Gaining traction in quantitative and real-time simulation circles.
*   **MCP (Model Context Protocol) Adoption**: Several frameworks now integrate MCP natively, treating external data sources (filesystems, databases, APIs) as first-class context providers rather than via bespoke tool definitions. This appears to be reducing boilerplate code for multi-source agents.
*   **"GPU-Resident" Agents**: Inspired by projects like **cudaclaw** (from our own fleet), there is increased experimentation with keeping agent state and small models entirely in GPU memory to avoid CPU-GPU transfer latency. The `llama.cpp` community has prototypes for persistent agent contexts in VRAM.
*   **Declarative Orchestration**: A move away from imperative Python scripts toward YAML/TOML-based declarative definitions of agent workflows, seen in forks of **LangGraph** and the new **FlowForge** project.

### 3. What's Cooling Off
*   **"Swarm" architectures** with dozens of homogeneous agents have less discussion, with focus returning to smaller, heterogeneous teams (2-5 agents) with clearly differentiated roles—aligning with our fleet's "senior agent" model.
*   **Pure prompt-engineered reasoning loops** are being supplemented or replaced by systems that integrate lightweight, trainable adapter layers (LoRAs) for role specialization, similar to our **plato-ensign** concept.

### 4. Fleet Relevance & Specific Observation
**Specific Observation:** The open-source trajectory validates key fleet architectural choices. The move toward **specialization (P2 optimization within a channel)** over generalist frameworks mirrors our doctrine. The growing interest in **MCP for context** and **GPU-residency for speed** suggests our investments in **holodeck-rust** (context-rich environments) and **cudaclaw** are aligned with external frontier development. However, the external community's focus on *declarative orchestration* is an area our fleet's codebases (plato-torch, fleet-simulator) do not yet emphasize, representing a potential channel for exploration (P1) or a negative space to avoid (P0) depending on our needs.

**Recommendation (P1 Channel):** Monitor the `hyperagent` Rust implementation for patterns that could inform `cudaclaw` optimization. Evaluate one declarative orchestration tool (e.g., FlowForge) to assess if it could simplify scenario definition in `fleet-simulator`.
