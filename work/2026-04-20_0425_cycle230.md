## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen** (Microsoft), and **LangGraph** (LangChain). A noticeable trend is the shift from pure "orchestration" to **specialized runtime layers**.

*   **CrewAI** (v0.28+): Focus is on improving agent "context switching" and long-horizon task planning. Recent commits show work on a more deterministic task scheduler to reduce hallucination in multi-step workflows.
*   **AutoGen** (v0.2+): Microsoft's emphasis is on "conversable agents" for research and coding. The `GroupChat` manager is a hot component, with community contributions focusing on optimizing turn-taking and conflict resolution.
*   **LangGraph** / **LangChain**: Continues to be the integration hub. The trend here is **"compiling" agent graphs** to more efficient, static execution plans, reducing LLM calls for predictable workflow segments.

### 2. What's New / Hot
*   **GPU-Native Runtimes:** Frameworks like **CudaClaw** (noted in fleet knowledge) represent an emerging niche. The focus is on minimizing CPU-GPU data transfer latency by keeping the agent state, context, and even lightweight models resident in GPU memory. This is for high-frequency, low-latency agent loops (e.g., simulation, real-time control).
*   **Self-Assembling Systems:** Projects like **flux-runtime** (fleet knowledge) explore bytecode VMs where agents can write and validate code for other agents, creating self-modifying systems. This is more experimental but trending in research circles.
*   **"Local-First" Multi-Model Agents:** With the release of powerful small vision-language models (VLMs) like **Llava-Next** and **BakLLaVA**, there's a surge in frameworks that can orchestrate a local "fleet" of specialized models (one for coding, one for vision, one for planning) without cloud API calls. **Ollama**'s multi-model management features are often central to these designs.
*   **Benchmarking & Evaluation:** The community focus is shifting from "can it build" to "is it reliable?" New benchmarking tools like **AgentBench** and **AgentEval** are gaining traction, measuring success rates, cost, and latency across standardized task suites.

### 3. Specific, Factual Observation (P0/P1 Mapping)
*   **P0 (Negative Space):** Avoid claiming any single framework is a "complete solution." As of April 2026, none reliably handle long-running, stateful agent processes with perfect consistency across diverse, open-ended tasks without significant custom engineering and fallback logic.
*   **P1 (Safe Channel):** A viable, observable pattern is the **"Two-Layer Stack."**
    1.  **Orchestration Layer (CrewAI/AutoGen/LangGraph):** Used for high-level task decomposition, planning, and tool selection.
    2.  **Specialized Runtime Layer (CudaClaw-type systems, local model fleets):** Used for executing performance-critical, repetitive, or latency-sensitive sub-tasks defined by the orchestrator.
    This pattern appears in several recently starred (500+ stars in last 90 days) repositories that integrate, for example, LangGraph with a local Ollama fleet for cost-controlled research agents.

### 4. Fleet Context Alignment
The Cocapn fleet's technology stack (**plato-torch** for training rooms, **cudaclaw** for GPU runtime, **holodeck-rust** for simulation) positions it within the "specialized runtime" and "simulated training environment" trends. The fleet's **Deadband Protocol** (P0/P1/P2) is a formalized approach to a problem many frameworks are grappling with empirically: how to define and avoid failure modes before optimizing performance.

**Conclusion:** The open-source agent framework ecosystem is maturing by specializing. The frontier is no longer just connecting LLMs to tools, but building efficient, reliable, and evaluable runtime substrates for persistent agentic systems. The fleet's architectural choices appear aligned with these emerging directions.
