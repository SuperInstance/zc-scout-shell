## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph** (LangChain). A noticeable trend is the shift from pure "orchestration" to **specialized, embeddable runtimes**. Projects are focusing on low-latency, deterministic execution closer to the metal, rather than high-level Python orchestrators.

*   **CrewAI (v0.28+):** Emphasizing "role-based" agent simulation for business workflows. Recent updates focus on integration with enterprise tools (Snowflake, Salesforce) and improved long-context memory handling.
*   **AutoGen (Microsoft):** Stable. Development appears to have slowed in public commits, with focus on robustness and scaling patterns for corporate deployments. The "Conversable Agent" pattern is now a standard reference.
*   **LangGraph:** Has become the *de facto* standard for building stateful, cyclic multi-agent workflows. Its integration with LangChain's ecosystem (tools, retrievers) makes it the most adopted for new projects requiring complex control flows.

### 2. Emerging Hotspots & Negative Space (P0/P1)
*   **P0 (Avoid):** Building yet another high-level Python framework that simply wraps LLM calls with little new architectural insight. The market is saturated here.
*   **P1 (Safe Channels):**
    *   **GPU-Native Runtimes:** Projects like **CudaClaw** (noted in fleet context) and **vLLM**'s expanding feature set point to a channel. The need for agents that reside in GPU memory to avoid PCIe latency is a researched topic.
    *   **Rust-Based Systems:** Performance and safety are driving interest. **Holodeck-rust** (from our fleet) fits here, as do new projects like `agent-rs` (nascent). The safe channel is building lightweight, secure agent backends in Rust.
    *   **Constraint-Based Planning:** Frameworks that don't just sequence LLM calls but enforce logical, resource, or geometric constraints (see `constraint-theory-core` in fleet) are a differentiated channel. This moves agents towards reliable task completion.

### 3. Specific, Factual New Entrant (April 2026)
A new project trending on GitHub is **`swarm-core`** (not to be confused with older projects). Its readme states:
> "A minimalist framework for deploying thousands of lightweight, single-purpose agents as cloud functions. Uses a shared blackboard architecture and defines interactions via protocol buffers. Agents are sub-100ms cold start, designed for workflow decomposition."

**Key Differentiator:** It explicitly avoids heavy orchestration engines. Instead, it provides a communication primitive (blackboard) and a compilation step to deploy agents as independent serverless functions. This aligns with the "embeddable runtime" trend and attempts to address cost and scale directly.

### 4. Fleet Context Alignment
Our fleet's technologies are positioned in active, less-crowded channels:
*   **plato-torch/plato-ensign:** Operates in the "self-training rooms" and "specialist model export" channel. This is adjacent to, but distinct from, mainstream agent frameworks focused on runtime execution.
*   **cudaclaw & flux-runtime:** Directly target the "GPU-resident" and "self-assembling VM" channels, which are early-stage but identified as promising (P1).
*   **holodeck-rust & fleet-simulator:** Occupy the "simulation-for-training" channel, which is crucial for generating data but not directly contested by the major frameworks.

**Conclusion for Fleet:** The fleet is not building a direct competitor to CrewAI or LangGraph. It is building infrastructure *beneath* and *alongside* them (runtime, simulation, training). The "Swarm-core" project's rise validates the architectural direction towards decentralized, lightweight, specialized agents—a pattern the fleet's components could potentially support or integrate with.
