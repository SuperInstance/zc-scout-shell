### 2. Specific Observation: The "Constraint" Pattern
A notable pattern emerging in adjacent projects (e.g., in robotics simulation and UI generation) is the formalization of **constraint solvers** as a core agent primitive. Instead of agents generating free-form actions, they propose actions that are then validated and "snapped" to a set of declarative constraints (physical rules, safety policies, API schemas).

**Fleet Relevance:**
This pattern is a concrete implementation of the **Deadband Protocol's P0 (negative space)**. The constraint solver defines the "rocks" – the invalid states. The agent's role (P1/P2) is to operate within the solution space the solver permits. The fleet's **constraint-theory-core** (Rust geometric snapping) is a specialized instance of this broader pattern. There is an opportunity to abstract this component into a more general **constraint engine** for agent action validation across multiple domains (code, narrative, shell commands).

### 3. Recommendation for Fleet Architecture (P1 Channel)
**Channel:** Integrate the **constraint-theory-core** (or a generalized successor) as a mandatory pre-commit layer for agent actions within the **plato-torch** training rooms. Before a tile is submitted, the proposed action/state could be checked against a room's defined constraint set (e.g., "cannot call `rm -rf /`", "must adhere to API YAML schema").

**Rationale:** This would harden the training loop, ensuring tiles are not just useful but also *safe* and *well-formed*, accelerating the creation of robust ensigns. It operationalizes P0 at the point of tile creation.
