### 1. Dominant Frameworks & Trends
The landscape remains anchored by three major paradigms:
*   **CrewAI:** Maintains strong focus on role-based, collaborative agents. Recent activity (last 30 days) shows increased emphasis on **long-running processes** and **inter-agent communication protocols**. The `crewai` repo shows 150+ commits in April, with new PRs for persistent memory and tool-calling refinements.
*   **LangGraph:** Continues as the de facto standard for building stateful, multi-step agentic workflows. The **"LangGraph Studio"** local development environment (released ~March 2026) is gaining adoption, shifting some prototyping from cloud notebooks to local IDEs.
*   **AutoGen:** Microsoft's framework shows steady development. A noticeable trend is integration with **smaller, specialized models** (e.g., 7B parameter class) for specific agent roles, rather than defaulting to large foundational models for all tasks.

### 2. Emerging Pattern: "GPU-Resident" or "In-Memory" Agents
A new pattern is emerging, distinct from the HTTP/API-based agent calls of the major frameworks. Projects like **`cudaclaw`** (noted in our fleet's key technologies) and similar external repos (e.g., `TensorRT-LLM` agent extensions) are experimenting with agents that live entirely within GPU memory. The claimed benefit is **sub-millisecond latency** for intra-agent communication, bypassing serialization and network overhead. This is currently a niche, systems-heavy approach.

### 3. Hot Topic: "Self-Improving" or "Self-Training" Agent Systems
There is increased discussion and prototyping around closed-loop systems where agents generate their own training data or fine-tuning tasks. This aligns with the fleet's **`plato-torch`** and **`plato-ensign`** concepts. External examples include:
*   **`OpenAI Evals`-style frameworks** being adapted for agent self-evaluation.
*   Research papers on **"LLM-as-a-Judge"** being operationalized into continuous training loops.
*   Several new (starred < 500) repos with names containing "self-rewarding" or "iterative fine-tuning" for agents.

### 4. Specific, Factual Observation (P0/P1 Mapping)
*   **P0 (Negative Space):** The "orchestrator" pattern—a single main LLM function-calling to sub-agents—is showing clear limitations in complex, long-horizon tasks. Debugging these chains is difficult. New frameworks are not heavily investing in this simple chain pattern.
*   **P1 (Safe Channel):** The safe channel for new agent system design appears to be **explicit state machines** (LangGraph's influence) and **shared, structured memory objects** (like CrewAI's context). Projects that adopt these patterns are receiving more consistent contributor engagement.
*   **P1 (Another Channel):** Integration with **traditional software metrics (APM, tracing)** is becoming a common feature request. The "agent monitoring" space is nascent but active, with projects like `langfuse` and `phoenix` adding specific agent lifecycle tracing.
