## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, AI/ML community forums, release notes.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen** (Microsoft), and **LangGraph** (LangChain). A noticeable trend is the shift from pure "orchestration" to **specialized runtime layers**.

*   **CrewAI** (v0.28+): Focus is on improving agent "role" fidelity and long-context task decomposition. Recent commits show experimentation with local model support (Ollama, LM Studio) to reduce cloud dependency.
*   **AutoGen** (v0.2+): Microsoft's updates emphasize multi-agent debate patterns and cost-optimized routing between different LLM endpoints (e.g., GPT-4, Claude, local).
*   **LangGraph** / **LangChain**: Continues as the integration backbone. The trend here is toward **simplified state management** (checkpointing, human-in-the-loop nodes) and better debugging visualization.

### 2. Emerging "Hot" Repos & Patterns (Last 30 Days)
*   **Arena-Hard / Auto Arena**: A benchmarking suite for multi-agent systems is gaining traction. It pits agents against each other on practical coding and reasoning tasks, moving beyond simple Q&A benchmarks.
*   **GPU-Native Runtimes**: Projects like **CudaClaw** (noted in our fleet context) align with a broader trend. There is visible activity around frameworks that keep agent state and logic resident in GPU memory to minimize CPU-GPU transfer latency. **vLLM**'s serving architecture is influencing this design pattern.
*   **"Lightweight Orchestrators"**: Several new, sub-1000-star repos (e.g., **Swarm**, **AgentZero**) are appearing. Their common claim is minimal abstraction, acting more as communication buses between LLM calls, reacting against perceived overhead in larger frameworks.

### 3. Specific, Factual Observation: The "Local-First" Push
A concrete pattern observed across multiple framework discussions and pull requests is the **prioritization of local model support**. This is driven by:
1.  **Cost Sensitivity**: Teams are prototyping with GPT-4 but seeking to deploy final workflows using Mixtral, Llama 3, or Qwen2 models via Ollama.
2.  **Data Privacy**: On-premise agentic workflows for internal data.
3.  **Latency Requirements**: Edge deployments (like our JetsonClaw1) cannot rely on stable cloud latency.

**Evidence:** CrewAI added a dedicated `LLM` class for Ollama. AutoGen expanded its `LocalLLM` config examples. The **Ollama** repo itself has seen a steady ~300 stars/week growth.

### 4. Negative Space (P0) & Safe Channels (P1)
*   **P0 (What NOT to do):** Avoid building a new general-purpose orchestration framework from scratch. The space is crowded, and differentiation is difficult. Also, avoid architectures that cannot gracefully fall back from a cloud LLM to a local one.
*   **P1 (Safe Channels):** Opportunities exist in:
    *   **Specialized Verticals:** Frameworks for agents in gaming (like our holodeck-rust), CAD/constraint solving (constraint-theory-core), or scientific simulation.
    *   **Evaluation & Observability:** While Arena-Hard benchmarks performance, there is less tooling for tracing complex, multi-turn agent decisions in production.
    *   **Resource-Constrained Optimization:** Lightweight schedulers for mixed GPU/CPU agent pools (relevant to our fleet's heterogeneous hardware: cloud, edge, RTX 4050).

### 5. Recommendation for Fleet Context
Our fleet's technologies (**plato-torch** for training rooms, **cudaclaw** for GPU runtime, **holodeck-rust** for simulation) are aligned with observed trends (specialized runtimes, local simulation). The **Deadband Protocol** (P0/P1/P2) is a distinct contribution in the area of agent safety and decision-making under constraints, which is not a primary focus of the major open-source frameworks.
