## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph**. A noticeable trend is the consolidation of features rather than the emergence of entirely new frameworks. Projects are focusing on **production readiness**, **observability**, and **cost control**.

*   **CrewAI (v0.28+):** Emphasizes "orchestration over code." Recent updates focus on **task-specific LLM routing** (e.g., sending a coding task to a coder agent model, a writing task to a writer model) and improved **inter-agent delegation** logic. Their "Process" abstraction (sequential, hierarchical, consensual) is a cited differentiator.
*   **AutoGen (Microsoft):** Continues to push the **multi-agent conversation** paradigm. Recent development appears focused on **tool integration robustness** and **streaming responses** for better user experience. The "GroupChat" manager with speaker selection is a core research-to-production feature.
*   **LangGraph (LangChain):** Gaining traction for developers who want **explicit control over agent state and flow**. Its strength is modeling complex, cyclic workflows (like a support bot that can loop back to gather more info). The "StateGraph" abstraction is often compared to building a finite-state machine for agents.

### 2. Emerging Pattern: The "Runtime" Layer
A notable development is the rise of projects focusing on the **underlying execution layer**, separate from the agent orchestration logic. This aligns with fleet technologies like `cudaclaw` and `flux-runtime`.

*   **Projects like `smolagents` and `aider`** (though aider is more coding-specific) exemplify a trend toward **lean, single-binary runtimes** that minimize dependencies and startup time.
*   The discussion is shifting from "how to define agents" to "how to **run agents efficiently, persistently, and with resource constraints**" (e.g., on a user's laptop, on an edge device like a Jetson). This is a P1/P2 channel the fleet's `cudaclaw` (GPU-resident runtime) already occupies.

### 3. Integration with Emerging Models
Frameworks are rapidly adding support for **new, high-performance open-weight models**.

*   **DeepSeek-Coder**, **Llama 3.1** variants (especially the 8B and 70B parameter sizes for different agent roles), and **Qwen 2.5** are frequently cited in configuration examples.
*   There is a visible push to **reduce reliance on GPT-4** due to cost and latency, creating a P1 channel for frameworks that demonstrate robust performance with open-source models.

### 4. "Hot" & Niche Areas
*   **Visual Agent Frameworks:** Increased activity around agents that can process screenshots/GUIs (e.g., **OpenDevin's** UI mode, projects like `screen-agent`). This connects to fleet interests in `holodeck-rust` and sentiment-aware environments.
*   **Code-First vs. Config-First:** A quiet tension exists. LangGraph is code-first. CrewAI leans config-first. The trend suggests **hybrid approaches** are winning for complex use cases, where a base workflow is configured but custom nodes/edges are coded.
*   **Evaluation & Benchmarking:** There is no consensus standard for evaluating multi-agent systems. Some projects are creating custom "agent arenas" (simulated environments), which is directly analogous to the fleet's `fleet-simulator` purpose.

### 5. Specific Observation for Fleet Context
The fleet's architectural concept of **"Rooms"** (from `plato-torch`) that accumulate **"tiles"** for training is a distinct pattern not observed in mainstream open-source frameworks. Most frameworks treat each agent session as ephemeral. The fleet's approach of **persistent, training-enabled environments** is a unique P1 channel. If the `plato-ensign` project successfully demonstrates exporting room experience as LoRA adapters, it could represent a novel contribution to the agent ecosystem—moving from **agent orchestration** to **agent evolution through situated experience**.

**P0 Note (Negative Space):** Avoid claiming any single framework is "the best." The optimal choice is highly dependent on use-case constraints (latency, cost, complexity, need for explicit control).

**P1 Channel Identified:** The intersection of **efficient agent runtimes** + **open-weight model support** + **persistent learning environments** appears less crowded than high-level orchestration frameworks.
