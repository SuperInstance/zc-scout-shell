## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Dominant Frameworks & Trends
The landscape remains anchored by **CrewAI**, **AutoGen**, and **LangGraph**, but the focus has shifted from pure orchestration to **specialization, observability, and cost control**.

*   **CrewAI (v0.28+):** Emphasizes "role-based" agent design with clear hierarchies (Manager, Worker). Recent updates focus on **tool memory** (agents remembering past tool outputs) and integration with **OpenAI-compatible local models** (LM Studio, Ollama) to reduce cloud API costs.
*   **AutoGen (Microsoft):** Continues to push **conversable agent** patterns for complex multi-agent workflows. The "GroupChat" with automated speaker selection is a common reference architecture. There is noticeable community activity around creating **custom agent types** for specific domains (e.g., data analysis, coding).
*   **LangGraph (LangChain):** Gaining traction for developers who want **fine-grained control** over agent state and execution flow using graph-based paradigms. Its strength is modeling cycles, interrupts, and human-in-the-loop processes, making it popular for building **long-running, stateful agent applications**.

### 2. Emerging Pattern: The "Runtime" & GPU-Resident Agents
A significant trend is the move towards **dedicated agent runtimes** that minimize latency. This aligns with fleet technology `cudaclaw`.

*   Projects like **`inference`** (from the creators of `llama.cpp`) are being adapted to serve as high-performance backends for agent loops.
*   There is research interest in **keeping the entire agent state (working memory, tools) GPU-resident** to avoid costly CPU-GPU transfers during iterative reasoning. This is not yet mainstream in the major open-source frameworks but is a frequent topic in research previews.

### 3. Hot New Entrant: Aider-style Coding Agents
While not a multi-agent framework per se, the pattern established by **`aider`** (an AI pair programmer that edits code in a local git repo) is influencing agent design. The key insight is **deep integration with the developer's environment** (git, linter, test runner) and **operating on a concrete file system**. Some new experimental frameworks are applying this "shell-as-environment" principle to broader multi-agent tasks.

### 4. Fleet Context Comparison
The Cocapn fleet's architecture, with its focus on **PLATO rooms for training**, **tile-based experience accumulation**, and the **Deadband Protocol (P0/P1/P2)**, represents a divergent path from the mainstream.

*   **Mainstream Focus:** Connecting to APIs (OpenAI, Anthropic), orchestrating chat completions, managing conversation history.
*   **Fleet Focus:** **Bootstrapping and self-training** via interaction rooms, exporting learned behavior as LoRA adapters (`plato-ensign`), and operating with a **git repo as the primary agent shell** (Zeroclaw concept). The fleet's `holodeck-rust` (sentiment-aware NPCs) and `fleet-simulator` are also specialized domains not directly addressed by the general-purpose frameworks.

### 5. Specific Factual Observation (P0 / Negative Space)
A common point of failure observed in community discussions is **agents hallucinating tool specifications or API parameters**. Many frameworks dynamically generate tool-calling schemas from code. If the docstring or function signature is ambiguous, the agent may call the tool incorrectly, leading to cascade failures. The **P0 lesson** here is: tool definitions for agents require stricter, more machine-parseable contracts than those for human developers.
