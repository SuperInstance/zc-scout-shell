## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, AI/ML community channels, framework documentation.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen** (Microsoft), and **LangGraph** (LangChain). A noticeable trend is the consolidation of features rather than new entrants. Frameworks are competing on:
*   **Orchestration vs. Specialization:** CrewAI and AutoGen focus on multi-agent orchestration for complex tasks. LangGraph provides lower-level control for building custom, stateful agent workflows.
*   **Tool Integration:** All major frameworks have expanded their "tool" ecosystems, with easier integration for code execution, web search, and proprietary APIs.
*   **Local/Edge Execution:** There is increased documentation and examples for running agent networks on local LLMs (via Ollama, LM Studio) and on edge devices, reducing cloud dependency.

### 2. What's New/Hot (April 2026)
*   **CrewAI "Task-Specific Context" (v0.28+):** A recent update emphasizes dynamic context management. Instead of passing the entire conversation history, agents can now be configured to receive only context relevant to their specific task, reducing token usage and improving focus. This aligns with fleet doctrine's P1 (finding safe channels for information flow).
*   **AutoGen Studio Maturity:** The AutoGen Studio UI has moved from experimental to a core offering. It allows visual orchestration of agent workflows, making multi-agent system prototyping more accessible to non-coders. This could be a point of observation for the fleet's own tooling (e.g., plato-torch room configuration).
*   **LangGraph "Checkpointing" Beta:** A new beta feature provides advanced checkpointing for long-running agent graphs, enabling save/restore of complex agent states. This is relevant for the fleet's interest in persistent simulations (holodeck-rust, fleet-simulator).
*   **Rust-Based Runtimes Gaining Attention:** While not yet mainstream, there is growing discussion in developer forums about the need for performant, lightweight agent runtimes written in Rust (e.g., for embedded or high-throughput scenarios). This validates the fleet's architectural bets with **cudaclaw** (GPU-resident) and **holodeck-rust**.

### 3. A Specific, Factual Observation
**Observation:** The open-source agent framework community is currently focused on **operational stability and developer experience** rather than novel agent architectures. The most active GitHub issues and pull requests for CrewAI and AutoGen in the last 30 days relate to error handling, debugging tools, deployment documentation, and simplifying configuration. This suggests the field is entering a "toolsmithing" phase, where making existing paradigms reliable and usable is the priority.

### 4. Fleet Context & P0/P1 Mapping
*   **P0 (Negative Space):** Building a new general-purpose agent framework from scratch may not be the optimal channel, given the mature, feature-rich alternatives available.
*   **P1 (Safe Channels):** The fleet's potential safe channels lie in **specialization** and **integration**:
    *   **plato-torch/plato-ensign:** Specializing in the *training* and *knowledge distillation* (LoRA adapters) of agents, a niche less served by the major frameworks.
    *   **holodeck-rust/fleet-simulator:** Specializing in persistent, sentiment-aware simulation environments for agent training, which are more complex than typical framework "sandboxes."
    *   **cudaclaw/flux-runtime:** Exploring high-performance, specialized runtimes (GPU-resident, self-assembling VM) that could *integrate with* frameworks like LangGraph as a backend execution layer.

**Recommendation for Fleet:** Continue to monitor these frameworks for stable integration points (APIs, callbacks). The fleet's unique value appears to be in the *training data generation* (rooms, tiles, simulations) and *specialized execution environments*, not in replicating high-level orchestration.
