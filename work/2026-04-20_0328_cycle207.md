## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph/LangChain**. However, the trend is shifting from monolithic "orchestrator" frameworks towards **lightweight, specialized runtimes** and **interoperability layers**.

*   **CrewAI** continues to emphasize role-based agent collaboration with structured task decomposition. Recent commits (last 7 days) show increased focus on integrating with **local LLMs via Ollama** and improving **agent memory persistence**.
*   **AutoGen** (Microsoft) is seeing development around **cost-optimized multi-agent conversations** and more granular **tool-calling controls**. The `ConversableAgent` pattern remains its core abstraction.
*   **LangGraph** has solidified its position for developers who want **explicit, graph-based control flow** over agentic workflows, often used to build custom frameworks on top.

### 2. Emerging Pattern: The "GPU-Resident" or "Edge-Native" Agent
A notable trend is the exploration of agents designed to live entirely within GPU memory or on edge devices, minimizing latency and cloud dependency. This aligns with fleet technologies like **cudaclaw**.

*   **Example:** Projects like **TensorRT-LLM Agent** prototypes and research around **vLLM** with continuous batching for agent loops are gaining attention. The goal is sub-second thought-loop latency.

### 3. Hot New Entrant & Research Focus (April 2026)
The most discussed new project in the last two weeks is **Aerie** (github.com/aerie-agent/aerie). It positions itself as a "**minimal, composable agent kernel**" rather than a framework.

*   **Key Differentiator:** It provides a simple, fast scheduler and message bus, allowing developers to bring their own LLM client, memory, and tools. It focuses solely on the **orchestration plumbing**. Early benchmarks show lower overhead compared to heavier frameworks.
*   **Community Sentiment:** Seen as a reaction to framework bloat. Developers are asking, "Do I need a full framework, or just a reliable way to run my agent functions in parallel with state?"

### 4. Integration with "Model Routers"
There is increased chatter about **dynamic model routing** within agent systems. Instead of hardcoding a model (e.g., `gpt-4`), agents are being configured to use a router that selects the best model for a subtask based on cost, latency, and capability.

*   **Open-source projects like OpenRouter** are being mentioned in agent design discussions. This creates a hybrid architecture where the agent framework handles logic, and an external service handles model selection.

### 5. Fleet-Specific Contextual Note

The mainstream focuses on **task completion**, while the fleet's research, based on provided knowledge, delves into **long-term interaction, training, and experience capture** (ensigns, ghost tiles). The fleet's work on **constraint-theory-core** (geometric snapping) is a unique specialization not widely seen in general agent frameworks.
