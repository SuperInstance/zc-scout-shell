### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen** (Microsoft), and **LangGraph** (LangChain). A noticeable trend is the shift from pure orchestration to **specialized, production-ready runtimes**. Frameworks are competing on inference speed, state management, and hardware integration.

*   **CrewAI (v0.28+):** Focused on improving agent "context switching" and long-horizon task planning. Recent commits show work on a persistent memory store that survives across agent sessions, addressing a common limitation.
*   **AutoGen (v0.2+):** Microsoft's framework is emphasizing **multi-modal agent teams** (text + vision) and tighter integration with Azure cloud services. Their research blog highlights work on reducing latency in multi-agent debates.
*   **LangGraph:** Has solidified its position as the low-level "assembly language" for agent workflows. Its strength is explicit state graphs and cyclicity. Many newer frameworks are building abstraction layers on top of LangGraph's core concepts.

### 2. Emerging & Niche Contenders
Several newer projects are gaining traction by solving specific problems:

*   **Agency (agency-py):** A minimalist framework promoting **single-file agents** and SSE/WebSocket streaming as a first-class citizen. It's attracting developers who want lightweight, real-time agent APIs without heavy dependencies.
*   **Phidata:** Gaining attention for its strong **structured outputs** (Pydantic) and built-in support for knowledge bases (RAG). Positions itself as the framework for building reliable, data-grounded agentic workflows.
*   **Model Context Protocol (MCP) Clients:** While not a framework itself, the open **MCP standard** (by Anthropic) for tool-use is influencing framework design. Projects building native MCP servers/clients (e.g., `mcp-clients`) are trending, indicating a move towards standardized, runtime-configurable tooling.

### 3. Technical Hotspots (What's "Hot")
*   **GPU-Resident Agents:** Exploration of keeping small LLMs (e.g., Phi-3, Qwen2.5) and the agent runtime itself in GPU memory to avoid PCIe latency. This aligns with fleet technology `cudaclaw`. See the `llama.cpp` server with continuous batching for agent use.
*   **Cost & Latency Optimization:** Frameworks are adding more sophisticated **token budgeting** and **early termination** features for chains of thought. The "cost" of an agent interaction is a primary metric.
*   **Evaluation & Benchmarks:** The lack of standard agent benchmarks (beyond simple tool-use) is a recognized gap. There's active discussion around creating open "AgentEval" suites, with preliminary work seen in the `agentbench` and `agentpilot` repos.

### 4. Fleet Context Comparison
The fleet's architecture, as described in `FLEET-CONTEXT.md`, has distinct parallels and divergences:
*   **Parallel:** The concept of **training rooms** (`plato-torch`) that accumulate experience ("tiles") mirrors the industry's focus on learning from agent trajectories. The "ensign" (LoRA adapter) is a specific implementation of the broader idea of **specializing base models** on agentic tasks.
*   **Divergence:** Most public frameworks are **cloud/CPU-first**. The fleet's deep focus on **edge GPU** (`JetsonClaw1`) and a dedicated **GPU-resident runtime** (`cudaclaw`) is a niche, performance-oriented path. The `holodeck-rust` MUD server as a training environment is also a unique approach compared to typical web-based simulators.
