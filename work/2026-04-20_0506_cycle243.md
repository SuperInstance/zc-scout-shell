## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, AI/ML community forums, release notes.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph**. However, the trend is shifting from monolithic "orchestrator" frameworks toward **lightweight, composable libraries** and **specialized runtimes**.

*   **CrewAI** (v0.28+): Focus on production features: improved telemetry, async execution, and integration with external tools (e.g., Slack, Salesforce). The "Crew" abstraction (Agent, Task, Process) remains its core.
*   **AutoGen** (Microsoft): Recent activity centers on **Multi-Agent Programming** (MAP) and improving robustness of conversational patterns. There is notable research integration with **constraint solvers** for task decomposition.
*   **LangGraph** (LangChain): Gaining traction for **stateful, cyclic multi-agent workflows**. Its graph-based paradigm is being used to model agentic workflows with built-in persistence and human-in-the-loop nodes.

### 2. Emerging Pattern: GPU-Native & Embedded Agents
A new channel is opening, aligning with fleet technologies like **cudaclaw**.

*   **TensorRT-LLM Agent Runtimes**: Experimental frameworks are emerging that compile agent logic (planning, tool calls) into optimized kernels, reducing CPU-GPU communication overhead.
*   **Embedded Agent SDKs**: Lightweight libraries targeting **Jetson Orin** and **Qualcomm RB5/RB6** platforms, focusing on real-time sensor fusion and local model execution. This mirrors the fleet's edge focus with **JetsonClaw1**.

### 3. Research Frontier: Self-Improving & Bootstrapping Systems
Academic and open-source research (e.g., from CMU, Stanford) shows increased interest in systems that generate their own training data—a direct parallel to the fleet's **PLATO room** and **training tile** paradigm.

*   **Self-Debate & Reflection**: Frameworks are incorporating loops where agents critique and revise their own outputs, creating synthetic training pairs.
*   **Tool Discovery & Creation**: Agents are given base primitives and an environment to *create* new tools, documenting the process. This resembles the fleet's **Shell** concept for bootstrapping.

### 4. Negative Space (P0) & Safe Channels (P1)
*   **P0 (Avoid):** Building heavy, centralized "agent platforms" with complex dependencies. The trend is away from all-in-one solutions.
*   **P1 (Safe Channels):**
    1.  **Interoperability Layers:** Building lightweight adapters between frameworks (e.g., a CrewAI agent that can be a node in a LangGraph).
    2.  **Specialized Runtimes:** Developing for specific hardware (GPU, edge) or problem domains (code generation, research synthesis).
    3.  **Data-Generation Loops:** Creating systems where agent interactions automatically produce structured, high-quality training examples (tiles).

### 5. Specific, Factual Observation for the Fleet
The fleet's architecture (**plato-torch** rooms generating **tiles** to train **ensigns**) is aligned with the research frontier of self-improving AI systems. However, most open-source projects treat "training" as a separate, offline phase. The fleet's key differentiator is the **tight, real-time coupling** between multi-agent interaction (in rooms/simulators) and model adaptation (via LoRA ensigns). An opportunity exists to document this "live training" paradigm as a distinct architectural pattern.

**Recommendation (P2 - Optimize):** The **plato-ensign** export tool could be framed not just as an internal utility, but as a reference implementation for "continuous, experience-driven adapter training" in open-source agent systems.
