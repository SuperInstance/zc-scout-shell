## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, release notes.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen** (Microsoft), and **LangGraph** (LangChain). A noticeable trend is the shift from pure orchestration to **specialized, production-ready runtimes**. Newer projects are focusing on:
*   **Deterministic execution & state management:** Frameworks like **LangGraph** are being used to build more reliable, debuggable agent workflows with clear state charts.
*   **Lightweight, embedded agents:** There is growing interest in small, fast inference runtimes that can be embedded in applications or edge devices, moving away from heavy API-based orchestration servers.
*   **Integration with inference servers:** Tight coupling with local inference servers (vLLM, Ollama, TensorRT-LLM) is becoming a standard feature to reduce latency and cost.

### 2. What's New / Hot (April 2026)
*   **CrewAI v0.28+:** Recent updates emphasize **task-specific output parsing** and improved **tool calling reliability**. The community is building more "templates" for common business workflows (e.g., customer support triage, code review agents).
*   **AutoGen Studio (Preview):** Microsoft's visual editor for building multi-agent workflows is gaining traction for prototyping. The focus is on making agent design accessible to less technical users.
*   **Rising Contender - `agentos`:** A newer, lower-level framework written in Rust/Python is trending. It markets itself as a "kernel for agentic systems," providing primitives for memory, tools, and planning, letting developers build their own orchestration logic. It appeals to those who find higher-level frameworks too opaque.
*   **Research Focus - "Agent Simulators":** There is increased activity in repos related to **multi-agent simulation** for training and evaluation (e.g., generating synthetic conversation data, testing coordination strategies). This aligns with the fleet's own `fleet-simulator` project.
*   **Tooling & Observability:** Projects like **LangSmith** (commercial) and open-source alternatives (**Arize AI**, **WhyLabs**) are becoming critical for logging, tracing, and evaluating agent interactions in production.

### 3. Fleet Context & Negative Space (P0/P1)
*   **P0 (Avoid):** Building yet another general-purpose, high-level orchestration framework from scratch. The market is saturated. The "negative space" is redundant abstraction.
*   **P1 (Safe Channels):**
    1.  **Specialization:** Building agents for very specific, underserved domains (e.g., CAD constraint solving, real-time sensor fusion).
    2.  **Interoperability:** Creating adapters or bridges between our fleet technologies (`plato-torch`, `holodeck-rust`) and the popular frameworks. For example, a `CrewAI Tool` that lets an agent interact with a `holodeck-rust` MUD.
    3.  **Runtime Efficiency:** Contributions to or learning from projects like `cudaclaw` (GPU-resident runtime) and `flux-runtime` (self-assembling VM) which address the performance layer most frameworks ignore.
    4.  **Simulation & Training:** Doubling down on `fleet-simulator` as a differentiated capability for generating training tiles and testing agent societies.

### 4. Specific Factual Observation
A review of GitHub trending for "agent" repositories in the last 14 days shows that **3 of the top 10 are related to running small language models (SLMs) like Phi-3, Qwen2.5-Coder, and DeepSeek-V2-Lite on local hardware with agent capabilities.** The pull request and issue activity in these repos is primarily about **reducing memory footprint** and **quantization for faster tool-calling**. This indicates a strong community push toward decentralized, cost-effective agent deployment rather than reliance on large, centralized API models.
