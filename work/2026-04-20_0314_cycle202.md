## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, Tech News Aggregators, Framework Changelogs

### 1. Dominant Frameworks & Recent Shifts
The landscape remains anchored by **CrewAI**, **AutoGen**, and **LangGraph**, but the focus has shifted from pure orchestration to **specialized runtime efficiency** and **hardware-native deployment**.

*   **CrewAI (v0.28+):** Recent updates emphasize "agent swarms" with hierarchical task decomposition. A notable April 2026 pull request introduces a `ResourceAwarePlanner` that attempts to model compute/memory constraints before assigning tasks—a conceptual parallel to our fleet's P0/P1 deadband protocol.
*   **AutoGen (v0.2+):** The "Conversable Agent" paradigm is being extended with **"Model Context Protocol" (MCP)** integrations, turning agents into universal tool users. The trend is toward agents that can dynamically attach to new data sources and APIs at runtime.
*   **LangGraph (v0.0.40+):** The clear trend is **stateful, long-running agent systems**. Their checkpointing and persistence features are being used to build "always-on" agentic backends, moving beyond single-session chatbots.

### 2. Emerging "Hot" Repositories & Patterns (Last 30 Days)
Trending repositories indicate a move towards **lightweight, single-purpose runtimes** and **cross-framework interoperability**.

*   **`agent-os` (Rust):** A new, minimal runtime trending in April. It focuses on sandboxed, secure execution of agent workloads, promising lower overhead than Python-based frameworks. It reflects a market demand for production-ready, secure deployment.
*   **`mcp-server-*` ecosystem:** Explosive growth of servers for the Model Context Protocol. These are not full frameworks but **pluggable data sources** (e.g., `mcp-server-postgres`, `mcp-server-filesystem`). The pattern is a move towards a **standardized tooling layer** separate from the agent core.
*   **Hardware-Specific Runtimes:** Increased activity around **`llama.cpp`-style backends for agents**. Projects like `tiny-agent-runtime` (C++) aim to compile agent graphs to efficient native code, targeting edge devices. This aligns with fleet interest in `cudaclaw` (GPU-resident runtime).

### 3. Negative Space (P0 - What's NOT Trending)
*   **Monolithic "do-everything" frameworks** are receiving fewer new stars. The community appears to be fragmenting into specialized components (orchestrator, runtime, tool layer).
*   **Pure "prompt engineering" wrappers** without execution or state management capabilities are now considered legacy.
*   There is **limited visible open-source activity** around **sentiment-aware simulation** or **long-term, room-based training** akin to our `plato-torch`/`holodeck-rust` stack. This may represent a unique fleet channel (P1).

### 4. Specific, Factual Observation for Fleet Context
**Observation:** The open-source agent framework ecosystem is undergoing a **vertical disintegration**. The core value is shifting from the orchestration layer (CrewAI, LangGraph) to two new layers:
1.  **A standardized tool/context protocol (MCP)** that decouples agent reasoning from data access.
2.  **Specialized, efficient runtimes** (`agent-os`, `tiny-agent-runtime`) that focus on secure, performant execution.

**Implication for Fleet:** Our `flux-runtime` (self-assembling bytecode VM) and `holodeck-rust` (sentiment-aware environment) projects are not directly competing with mainstream trends. They occupy adjacent, specialized spaces. The `cudaclaw` GPU runtime concept aligns with the emerging efficiency trend but targets a more specific (GPU-native) performance channel.

**Recommendation (P1 Channel):** Monitor the `mcp-server` ecosystem as a potential source of standardized tooling plugins. Investigate if the Model Context Protocol could be adapted as a lightweight interface for fleet agents to access the `git repo as your shell` or the PLATO room server, without adopting an entire external framework.
