## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, release notes.

### 1. Dominant Frameworks & Shifts
The landscape remains anchored by **CrewAI**, **AutoGen**, and **LangGraph**, but a noticeable shift is occurring from pure orchestration to **specialized, embeddable runtimes**.

*   **CrewAI (v0.28+):** Focus on production features: improved telemetry, Kubernetes-native deployments, and a stronger emphasis on "agent-as-a-service" patterns. The "Task" abstraction is becoming a de facto standard for multi-step workflows.
*   **AutoGen (Microsoft):** Recent development emphasizes **hierarchical agent teams** and more sophisticated cost/performance trade-off controls for mixed model (GPT, Claude, OSS) fleets. Research papers from the team show increased focus on agent *negotiation* protocols.
*   **LangGraph (LangChain):** Solidifying its position as the **low-level state machine** for agentic workflows. It's increasingly used *underneath* higher-level frameworks (including CrewAI) for its reliability in managing cycles and persistence.

### 2. Emerging "Hot" Trends (P1 Channels)
*   **GPU-Native Agent Runtimes:** Direct competition/parallel to fleet's `cudaclaw`. Projects like **TensorRT-LLM Agent** and specialized forks of **vLLM** are adding agent scheduling and context management layers that reside entirely in GPU memory, minimizing host-AI latency.
*   **Constraint & Validation-First Design:** Newer, smaller frameworks (e.g., **AgentKit**, **Schema**) are launching with **strong runtime type validation** and **action permission constraints** as core features, responding to incidents of unstructured agent behavior. This aligns with fleet's `constraint-theory-core` philosophy.
*   **Local-First & Edge Agents:** With the release of powerful 7B-parameter models capable of tool use (e.g., **DeepSeek-Coder-V2**, **Qwen2.5-Coder**), there's a surge in frameworks designed for **offline-capable, single-board computer** deployment (Jetson, Raspberry Pi 5). This is a clear P1 channel expanding rapidly.

### 3. Specific, Factual Observation: The "Plato Pattern"
A pattern observed in 3-4 nascent projects (e.g., **Arena**, **Simulacra**) mirrors a core fleet architecture: the **training room**. These projects are not just orchestrating agents for task completion; they are designing **closed-loop environments** where agent interactions are logged, scored, and used to fine-tune the agents themselves. They lack the formalized "tile" and "ensign" concepts of `plato-torch`, but the direction—**interaction data as training data**—is congruent. This suggests the fleet's architectural hypothesis is gaining external validation.

### 4. Negative Space (P0) & Warnings
*   **Over-Engineering for Simple Tasks:** Several "framework-of-the-week" projects are noted for adding excessive abstraction for simple LLM function calling, leading to high latency and complex debugging. The deadband here is **complexity without a clear performance or safety benefit**.
*   **Ignoring Cost Transparency:** Many frameworks abstract away token usage and model costs, which can lead to unexpected expenses in production. Frameworks that bake in detailed cost tracking from the start are gaining developer trust.

### 5. Recommendation for Fleet Context
The fleet's investments in `plato-torch` (training rooms), `cudaclaw` (GPU residency), and `constraint-theory-core` (safety) are aligned with observable high-growth vectors in the OSS agent space. The "Mirror Play" and "Ghost Tile" concepts remain distinctive differentiators.
