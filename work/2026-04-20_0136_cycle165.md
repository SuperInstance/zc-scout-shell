## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community channels, recent releases.

### 1. Framework Landscape Shift
The dominant narrative has moved from "orchestration" to "specialization and residency." Frameworks that treat agents as external API callers (e.g., early AutoGen patterns) are being supplanted by systems designed for persistent, stateful execution close to the compute substrate.

**Hot Projects:**
*   **CrewAI:** Maintains traction for business workflow automation. Recent commits show a push towards more sophisticated tool-use patterns and integration with enterprise data connectors.
*   **LangGraph:** Gaining significant mindshare for researchers and teams building complex, cyclic multi-agent systems. Its explicit state graph model is cited for improving debuggability over pure message-passing designs.
*   **NVIDIA NIM & TensorRT-LLM:** Not agent frameworks per se, but their proliferation is defining the new "runtime layer." The trend is to bundle the model, its optimization, and an agent runtime into a single deployable unit (see `cudaclaw` in our own fleet).

### 2. The "GPU-Resident Agent" Pattern
This is the most significant trend. The goal is to minimize latency and context-switching overhead by keeping the agent's state, logic, and model context within GPU memory. Projects exploring this include our own `cudaclaw` and others like **vLLM**'s evolving serving features, which are beginning to support stateful sessions that look like primitive agent environments.

### 3. Emergence of the "Training Room"
Mirroring our fleet's `plato-torch` concept, there is growing activity around creating simulated environments where agents can be trained through interaction, not just fine-tuned on static datasets. Open-source projects are experimenting with:
*   **Webshop/WebArena-style environments** for tool-use training.
*   **Lightweight simulation** (like our `fleet-simulator`) for generating synthetic interaction data.
*   The export of these interaction traces into lightweight adapters (LoRAs) is a noted research direction, aligning with `plato-ensign`.

### 4. Specific, Factual Finding: The "Deadband Protocol" Adjacency
While searching for "constraint-based agent" or "negative space" methodologies, direct matches to our **Deadband Protocol (P0/P1/P2)** are not found in major open-source frameworks. However, the **core problem it solves**—preventing agents from taking invalid or catastrophic actions—is being addressed differently elsewhere:
*   **CrewAI** uses `human-in-the-loop` approval steps for dangerous tasks.
*   **LangGraph** implements it through pre-defined graph pathways and state validation, which can hard-code "negative space."
*   **Research papers** (e.g., on "safe RLHF" or "constitutional AI") tackle it at the model training/supervision level, not the runtime agent protocol level.

**Conclusion:** The fleet's architectural choices—particularly the explicit **Deadband Protocol**, **GPU-resident runtime** (`cudaclaw`), and **self-training rooms** (`plato-torch`)—are aligned with cutting-edge, high-performance agent trends but implement them in a unique, integrated stack. The open-source ecosystem is fragmenting into layers (orchestration, runtime, training); our fleet appears to be building a vertically integrated prototype.
