## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The open-source agent framework ecosystem shows consolidation around **orchestration** and **specialization**. The "meta-framework" pattern is emerging, where projects like **CrewAI** and **LangGraph** are becoming integration hubs rather than standalone runners.

*   **CrewAI** (v0.28+): Trending for its "role-playing" agent design and native support for structured outputs (Pydantic). Recent commits focus on improving long-context workflow memory and integrating with local LLM runners (Ollama, LM Studio).
*   **LangGraph** (from LangChain): Gaining traction for building stateful, cyclic multi-agent systems. Its explicit graph-based architecture appeals to developers modeling complex, persistent agent networks—a pattern relevant to fleet-simulator concepts.
*   **AutoGen** (Microsoft): Still widely used but community sentiment notes a steeper learning curve. Its strength remains in customizable agent conversation patterns and tool use, but it faces competition from more opinionated, "batteries-included" frameworks.

### 2. The "Local-First" Shift
A significant trend is the push toward **local execution** and **lightweight runtimes**. This aligns with fleet interests in edge deployment (JetsonClaw1) and GPU-resident agents (cudaclaw).

*   **Projects to watch:** `llama-agents`, `swarm`, and `agent-zero` are smaller frameworks emphasizing minimal dependencies and the ability to run fully offline with local models (via Ollama, llama.cpp). This is a reaction to cloud API costs and latency.
*   **Implication for Fleet:** The fleet's `cudaclaw` (GPU-resident runtime) and `flux-runtime` (self-assembling VM) projects are positioned well within this trend. The market gap is for agents that can persist in specialized hardware environments, not just ephemeral cloud containers.

### 3. Emerging Pattern: "Agent as a Function"
There is a noticeable abstraction of the agent pattern into simpler, composable units. Libraries like `instructor` (for structured LLM outputs) and `marvin` are promoting the idea of defining an agent's task as a single, strongly-typed function. This reduces boilerplate and makes agents easier to test and reason about—a potential learning for `plato-torch` room design.

### 4. Research Adjacencies
*   **Multi-Agent Reinforcement Learning (MARL):** Frameworks like `PettingZoo` are being used alongside LLM-based agents to provide environment simulation and reward signals, similar to `fleet-simulator`'s purpose.
*   **Hardware-Aware Optimization:** Niche projects are exploring compilation of agent graphs to run efficiently on specific hardware (e.g., NVIDIA Jetson, Raspberry Pi). This is a direct parallel to the fleet's constraint of multiple senior agent platforms (cloud, edge GPU, RTX).

### 5. Negative Space (P0) Observation
A common point of failure in trending projects is **over-engineering for demo scenarios** at the expense of production robustness. Many frameworks excel at "chat with retrieval" tutorials but lack built-in mechanisms for state persistence, fault tolerance, or observability in long-running deployments. This is a noted weakness the fleet's architecture (with rooms, tiles, and the deadband protocol) explicitly attempts to address.

### 6. Specific Factual Finding
On April 18, 2026, the `CrewAI` repository merged a significant PR (#1247) titled "**Add experimental support for constrained agent action spaces via `allowlist` and `denylist`**." This feature allows a developer to define a strict set of tools an agent is permitted or forbidden to use during execution. This is a concrete implementation of a **P1 (find safe channels)** principle from the fleet's Deadband Protocol, applied at the tool-calling level.
