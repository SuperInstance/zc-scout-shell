## Output: Scout Report — April 2026 Open-Source Agent Framework Landscape

**Observation Period:** April 10–19, 2026  
**Source:** GitHub trending, major AI/ML community hubs, release notes.

### 1. Trending Frameworks & Shifts
- **CrewAI** (v0.28+): Strong focus on multi-agent orchestration with improved task delegation and memory streams. Recent commits show integration with local LLMs via Ollama, aligning with fleet's edge-GPU (Jetson) interests.
- **AutoGen** (Microsoft): Stable. Emphasis remains on conversational agent programming. Not seeing major architectural shifts; community activity is in specialized agent patterns (coding, research).
- **LangGraph** (LangChain): Gaining traction for *stateful*, cyclic multi-agent workflows. The explicit graph-based control flow is a noted contrast to CrewAI's hierarchical approach. This mirrors fleet's interest in structured agent interaction (e.g., holodeck-rust NPC sentiment cycles).
- **New Entrant — `swarm` framework:** Early-stage Rust-based framework for agent swarms with compile-time safety guarantees. Low stars but aligns with fleet's Rust stack (constraint-theory-core, holodeck-rust). Worth monitoring.

### 2. Emerging Pattern: Local-First & GPU-Resident Agents
- Several trending repos (e.g., `local-llm-agent`, `gpu-agent-runtime`) focus on minimizing cloud dependency and maximizing GPU memory residency.
- This directly parallels **cudaclaw** (GPU-resident agent runtime) in the fleet stack. The trend validates the fleet's architectural bet.
- Observed techniques: model sharding for multi-GPU, persistent agent state in VRAM, CUDA graph optimizations for agent step functions.

### 3. Fleet-Relevant Gap: Training Data from Agent Interactions
- Most frameworks treat agents as *inference-only* — they execute tasks but do not systematically capture their interactions as training data.
- The fleet's **plato-torch** (training rooms) and **fleet-simulator** (simulation for training data) represent a distinct approach: agents *produce* the tiles that train subsequent specialist models (ensigns).
- This gap is a potential fleet advantage. No trending project explicitly links agent runtime to a self-training loop with LoRA export (plato-ensign).

### 4. Specific Observation: The "Deadband Protocol" as a Differentiator
- The fleet's **Deadband Protocol** (P0 negative space, P1 safe channels, P2 optimization) is a structured constraint-satisfaction approach.
- In the open-source landscape, constraint handling is typically ad-hoc (prompt engineering, validation functions). No framework found implements a geometric snapping or negative-space mapping paradigm like **constraint-theory-core**.
- This could be a unique fleet contribution if documented as a reusable pattern.

### 5. Recommendation for Fleet Development
- **P0 (Negative Space):** Avoid replicating generic orchestration features already mature in CrewAI/LangGraph.
- **P1 (Safe Channel):** Focus development on the unique stack: cudaclaw (GPU residency) + plato-torch (self-training rooms) + constraint-theory (geometric constraints). This is an open niche.
- **P2 (Optimize):** Consider exporting fleet patterns as lightweight libraries (e.g., a "deadband" Python decorator, a GPU-resident agent base class) to attract community contributors.
