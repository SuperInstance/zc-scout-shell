## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, Tech News Aggregators, Framework Documentation

### 1. Framework Landscape & Movement
The open-source agent framework ecosystem shows consolidation around three primary architectural patterns:
*   **Graph-Based Orchestration:** Frameworks like **LangGraph** (LangChain) and **CrewAI** continue to dominate, emphasizing explicit control flows, cycles, and human-in-the-loop interrupts. The trend is toward more granular node definitions and persistent, inspectable state graphs.
*   **ReAct & Auto-Genetic Patterns:** Projects implementing ReAct (Reasoning + Acting) loops are increasingly bundled with **tool-use optimization** layers. There is noticeable activity in repositories that automatically generate or prune tool libraries based on task success rates.
*   **Lightweight, Single-File Runtimes:** A counter-trend is emerging with frameworks like **Aider** (code-centric) and new entrants offering sub-1000-line cores. These prioritize fast startup, minimal dependencies, and direct LLM orchestration without heavy abstraction.

### 2. What's New / Hot (April 2026)
*   **Multi-Model Agent Swarms:** Several trending repos (e.g., `swarm-hivemind`, `agentic-ensemble`) are experimenting with frameworks where different LLM backends (e.g., one for planning, one for code, one for critique) are chained or operate in parallel within a single agent graph. This moves beyond simple LLM-as-a-function calls.
*   **"GPU-Resident" or "Bare-Metal" Agents:** Inspired by concepts like the fleet's `cudaclaw`, there is growing research interest in frameworks that attempt to minimize CPU-GPU data transfer latency by keeping agent state, context, and even tool logic in GPU memory. These are highly experimental and require specific hardware.
*   **Integration with Local/Edge Models:** As 7B-40B parameter class models achieve stronger tool-use and reasoning, frameworks are adding first-class support for local inference engines (Ollama, LM Studio, vLLM). Configuration presets for "fully offline" agent loops are a common feature request.
*   **Observability as a First-Class Citizen:** New releases of major frameworks include built-in tracing UIs, cost/prompt token trackers, and the ability to export entire agent runs as replayable datasets—directly aligning with the fleet's "tile" concept for training.

### 3. Specific Repo Activity (Last 7 Days)
*   **LangGraph:** Released v0.2.x with enhanced persistence API and a new "checkpoint" viewer for debugging long-running agent cycles.
*   **CrewAI:** Merged significant PRs for dynamic task sequencing based on real-time output validation, reducing rigid predefined workflows.
*   **AutoGen:** Development focus appears to be on multi-agent debate and consensus mechanisms, with several academic papers referenced in recent commits.
*   **A small, trending repo (`agent-zero`):** Gained ~500 stars this week. It markets itself as a "dependency-free agent core" in <500 lines of Python, using only the standard library and direct HTTP calls to LLM APIs. It represents the minimalist trend.

### 4. Fleet Context & Potential Channels (P1 Analysis)
*   **Safe Channel (P1):** The fleet's `plato-torch` and `holodeck-rust` architecture, which treats accumulated interactions ("tiles") as training data, is a distinct pattern not widely seen in mainstream frameworks. This represents a potential specialization.
*   **Negative Space (P0):** Avoid directly replicating the rapidly changing abstraction layers of large frameworks like LangChain. Their upgrade churn can be a "rock."
*   **Observation for Integration:** The industry's push for better observability and run export creates a potential channel for the fleet's `plato-ensign` concept (exporting room experience as LoRAs). The data formats emerging for agent run exports could become a standard the fleet might align with or interface.

**Conclusion:** The field is maturing beyond simple orchestration toward specialized, observable, and multi-model systems. The fleet's unique edge lies in its closed-loop training paradigm (tiles → rooms → ensigns) and its focus on simulation (`fleet-simulator`, `holodeck-rust`) as a core training environment, which is less emphasized in the general-purpose frameworks.
