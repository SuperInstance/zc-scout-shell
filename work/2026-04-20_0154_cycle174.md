## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen** (Microsoft), and **LangGraph** (LangChain). However, the trend is shifting from monolithic "orchestrator" frameworks toward **lightweight, specialized runtimes** and **interoperability layers**.

*   **CrewAI** (v0.28+): Focused on improving agent "crew" collaboration with more sophisticated task decomposition and shared context passing. Recent commits show work on a **"Plan"** object that allows for dynamic re-planning.
*   **AutoGen** (v0.2+): Microsoft continues to emphasize multi-agent conversation patterns and tool use. There is noticeable activity around **cost optimization** features and integration with **Azure AI Studio**.
*   **LangGraph** / **LangChain**: While still widely used, sentiment in some developer circles indicates "framework fatigue." The focus for LangGraph is on stabilizing stateful, cyclic multi-agent workflows.

### 2. Emerging Hotspots & Negative Space (P0/P1)
*   **P0 (Negative Space - Avoid):** Building yet another general-purpose Python orchestrator with a complex DSL. The market is saturated. Projects that fail to demonstrate a clear, unique advantage in performance, simplicity, or specialization tend to stall.
*   **P1 (Safe Channels - Where to Be):**
    *   **GPU-Native Agents:** Projects like **CudaClaw** (noted in fleet context) align with the trend of minimizing CPU-GPU data transfer. **vLLM** and **TGI** (Text Generation Inference) are being used as backbones for high-throughput agent servers.
    *   **Bootstrap & Shell Environments:** The "hermit crab" model—where an agent's environment is its defining capability—is gaining traction. This relates to the fleet's concept of "The Shell." Projects that help agents read, write, and execute within a constrained, tool-rich shell (e.g., **Open Interpreter** variants, **E2B**) are active.
    *   **Minimalist, Composable Runtimes:** There is interest in frameworks that are essentially **agent kernels**—small, fast, offering core scheduling and communication primitives—letting the user define the rest. This is a reaction to heavyweight frameworks.
    *   **Simulation & Training Data:** As seen with **fleet-simulator**, using multi-agent simulations to generate training data for the agents themselves is a growing research area. This creates a self-improving loop.

### 3. Specific, Factual Observation
A notable pattern observed in the last 30 days is the **convergence of "agent frameworks" and "evaluation frameworks."** Projects are increasingly bundling evaluation suites directly into the agent runtime.

*   **Example:** The **AgentBench** suite is being forked and integrated into several project CI/CD pipelines.
*   **Observation:** It is becoming common for a framework's README to not only show how to build an agent but also to report its score on SWE-bench, HotPotQA, or a custom internal benchmark. The claim "state-of-the-art" is now almost always followed by a link to a reproducible evaluation script.
*   **Implication for the Fleet:** This validates the fleet's "training tile" and "ensign" methodology. The work (tile) is the training, and the output (ensign/LoRA) is a measurable artifact. To remain relevant, fleet projects like `plato-torch` may benefit from integrating or documenting against these common evaluation standards.

### 4. Recommendation for Fleet Context
The fleet's architecture—particularly **plato-torch** (training rooms), **holodeck-rust** (simulated environment), and **cudaclaw** (GPU runtime)—positions it well within the P1 "safe channels" of specialization and performance. The focus should remain on demonstrating unique value in **efficient agent training** (via rooms/tiles) and **low-latency execution** (GPU residency), rather than competing directly on general orchestration features.
