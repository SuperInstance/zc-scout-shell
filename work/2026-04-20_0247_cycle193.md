## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, release notes.

### 1. Framework Consolidation & Specialization
The landscape shows a move away from general-purpose "orchestrator" frameworks. Projects are specializing:
*   **CrewAI** and **LangGraph** remain popular for multi-agent workflows with clear state machines.
*   **AutoGen** (Microsoft) is seeing increased adoption in enterprise RAG and coding agent scenarios, but its complexity is noted as a barrier.
*   **New entrants** like **Aider** (code-focused) and **OpenDevin** (attempting open-source SWE-bench agents) are gaining traction in specific verticals. Their success appears tied to narrowly defined, repeatable tasks.

### 2. The "Runtime" Shift
A notable trend is the focus on lightweight, deterministic **runtimes** over heavy orchestrators.
*   **Cline** (by Codium) and **Continue** are examples of editor-native agents that act as fast, context-aware copilots, not standalone systems.
*   The fleet's own **cudaclaw** (GPU-resident runtime) and **flux-runtime** (self-assembling VM) align with this trend toward efficient, embedded execution.

### 3. Evaluation & Benchmarks Become Critical
With many frameworks claiming capability, the community focus has shifted to evaluation.
*   **SWE-bench**, **AgentBench**, and **MLAgentBench** are standard test suites. A framework's published scores on these are a primary adoption driver.
*   There is skepticism toward demos without reproducible benchmark results. The "simulation" approach seen in **fleet-simulator** is relevant here.

### 4. Open-Source Model Integration
Frameworks are rapidly integrating new open-source models.
*   Support for **DeepSeek-Coder**, **Qwen2.5-Coder**, and small, fine-tuned models (7B-34B parameter range) is common.
*   The pattern is to use a large model (e.g., GPT-4, Claude 3.5) as a "planner" or "verifier" and smaller, faster OSS models for execution. This matches the fleet's potential use of **plato-ensign** LoRA adapters for specialized tasks.

### 5. Specific Hot Repo (Example Observation)
**Repository:** `continuedev/continue`
**Trend:** Rising stars on GitHub (Python).
**Observation:** Continue is an open-source autopilot for VS Code & JetBrains. Its April 2026 activity shows a major push toward **local, offline operation** using Ollama and LM Studio, reducing latency and cost. This aligns with the broader trend of moving inference to the edge, a domain relevant to fleet agent **JetsonClaw1**.

### P0 / P1 / P2 Analysis for the Fleet
*   **P0 (Negative Space):** Avoid building another general-purpose, cloud-only agent orchestrator. The market is saturated. Avoid frameworks that lack transparent benchmarking.
*   **P1 (Safe Channels):** Specialized runtimes (like cudaclaw), simulation-based training (fleet-simulator), and edge deployment (Jetson) are clear differentiators. Integrating with fast, local OSS models is a viable path.
*   **P2 (Optimization):** Within the runtime/specialization channel, optimizing for latency, memory footprint, and deterministic execution (e.g., via **constraint-theory-core** for structured output) could provide an advantage.

**Conclusion:** The open-source agent framework ecosystem in April 2026 values specialization, verifiable performance, and efficient local execution over broad orchestration. The fleet's existing architectural pieces (specialist ensigns, GPU runtime, simulation) are aligned with these trends.
