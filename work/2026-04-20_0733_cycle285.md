## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph**. However, the trend is shifting from monolithic "orchestrator" frameworks toward **lightweight, composable libraries** that act as "agent primitives."

*   **CrewAI** (v0.28+): Focus on production features. Recent updates emphasize **interruptibility** and **contextual memory** that persists across sessions, moving beyond single-task workflows.
*   **AutoGen** (Microsoft): Strong research lineage. The community build, **AutoGen Studio**, provides a low-code UI, but adoption seems split between academic use and enterprise pilots.
*   **LangGraph** (LangChain): Gaining traction for developers who want **explicit control over state machines**. Its appeal is the transparent, debuggable flow defined as a graph. It's often cited as a "building block" rather than a full-stack solution.

### 2. Emerging Pattern: The "Runtime" Layer
A new category is emerging, separate from orchestration: **dedicated agent runtimes**. These handle the low-level execution environment, not the logic.
*   **Example:** **Cudaclaw** (noted in fleet context) and projects like **Axia** propose keeping the agent's state and tools resident in GPU memory to reduce latency. This aligns with a broader industry exploration of **persistent, stateful agents** versus stateless, per-request invocations.

### 3. Hot Topic: Multi-Model & Specialized "Ensigns"
The conversation is moving beyond "which LLM?" to **dynamic model routing**. Frameworks are adding support for:
*   **Cost/performance routing:** Sending simple tasks to smaller, cheaper models (e.g., Phi-3, Qwen2.5) and complex reasoning to larger models.
*   **Tool-specific routing:** Using a vision model for image analysis within an agent's step, then a code model for the next step. This mirrors the fleet's **plato-ensign** concept of exporting room experience into specialized LoRA adapters.

### 4. Negative Space (P0) & Safe Channels (P1)
*   **P0 (Avoid):** Building a single, giant "super-agent" monolith. The consensus is this becomes brittle and hard to debug. Over-reliance on a single framework's proprietary abstraction is also seen as a risk.
*   **P1 (Safe Channels):**
    1.  Using **LangGraph** or similar for core, auditable control flow.
    2.  Leveraging **lightweight libraries** for specific capabilities (e.g., `guidance` for constrained generation).
    3.  Designing for **model interchangeability** at the component level.
    4.  Exploring **runtime optimizations** (like GPU residency) for latency-sensitive applications.

### 5. Fleet Context Alignment
The fleet's architecture appears aligned with observed trends:
*   **plato-torch rooms** as specialized training environments map to the trend of creating **specialized agent components**.
*   The **ensign** export concept is a concrete implementation of the "specialized model" trend.
*   **Holodeck-rust** (sentiment-aware NPCs) touches on the growing interest in **long-running, stateful agent simulations** for training, as seen in projects like **fleet-simulator**.

**Specific Factual Observation:** The GitHub trending repository `agent-zero` (a new entry last week) demonstrates the lightweight primitive trend. It provides under 500 lines of code to spawn concurrent agent workers with shared context, explicitly positioning itself as "a screwdriver, not a workshop." This suggests developer preference for composability over integrated suites.
