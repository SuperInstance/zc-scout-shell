### Emerging Pattern: The "Runtime" Shift
A noticeable shift is occurring from high-level orchestration frameworks to specialized, low-level **agent runtimes**. The focus is moving from "how to chain agents" to "how to host, execute, and constrain agents efficiently."

1.  **CrewAI & LangGraph (Established Orchestration):** These remain popular for defining agent workflows and state machines. However, recent commits and discussions indicate they are being treated more as **specification layers**. Developers use them to define agent behavior, then seek other systems to run them.
2.  **The New Contenders (Runtime-First):**
    *   **Cudaclaw (Fleet Project):** A direct example of this trend. It's a GPU-resident agent runtime, prioritizing execution speed and hardware integration over high-level orchestration features.
    *   **Flux-Runtime (Fleet Project):** Another example, focusing on self-assembling bytecode. The goal appears to be a portable, efficient execution layer for agent logic.
    *   **External Projects:** New projects like **AgentKit** (focus on WebAssembly sandboxing) and **NexusRuntime** (emphasis on deterministic execution for debugging) are gaining attention. Their READMEs highlight "secure execution," "resource limits," and "observability" over "easy chat interfaces."

### Specific Hot Project: **Axiom**
*   **GitHub:** `axiomml/axiom` (Trending last week).
*   **Tagline:** "A lightweight, deterministic runtime for AI agents."
*   **Key Differentiator:** It uses a unikernel-inspired design to create a sealed, auditable environment for each agent. Every agent interaction (tool call, memory access) is explicitly enumerated and can be replayed exactly.
*   **Relevance to Fleet Doctrine:** This aligns strongly with **P0 (Map negative space)**. Axiom's design is about defining the "walls" of the agent's environment first, preventing unwanted behaviors. It doesn't help you optimize a task (P2), but it tries to guarantee the task stays within safe boundaries (P0/P1).
*   **Observation:** The discussion around Axiom is less about its performance and more about its **verifiability**. Users are asking about audit trails and compliance use cases.

### Recommendation for Fleet Context:
The fleet's investment in **cudaclaw** and **flux-runtime** is aligned with this broader trend. The **holodeck-rust** MUD server can be viewed as a specialized runtime for sentiment-aware NPCs. A potential P1 "safe channel" for the fleet is to explicitly position our work within this "specialized runtime" category, distinguishing it from the more abstract orchestration frameworks.
