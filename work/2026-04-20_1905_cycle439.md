## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen** (Microsoft), and **LangGraph** (LangChain). A noticeable trend is the consolidation of features rather than new framework proliferation. The "multi-agent" paradigm is now table stakes; the current focus is on **orchestration efficiency**, **state persistence**, and **cost-aware execution**.

### 2. What's New/Hot (April 2026)
*   **CrewAI v0.28+**: Introduced "Task Sandboxing" — isolated execution contexts for agents to prevent prompt leakage and allow rollback. Their recent blog highlights benchmarking against a synthetic "Boardroom" simulation, showing a ~15% reduction in token waste via improved delegation logic.
*   **AutoGen Studio** (the UI layer) now has a visual workflow builder that exports to code. The hot topic in their Discord is the experimental "Agent Memory Bank," a vector store that persists across sessions for long-running assistant agents.
*   **LangGraph** is pushing its **"Checkpointer"** system hard, enabling long-running, fault-tolerant agent cycles. This is particularly relevant for production deployments where agents run for hours/days. Their "StateGraph" abstraction is being adopted by several newer, niche frameworks.
*   **Emerging Niche:** **Swarm** frameworks (e.g., `swarm-core`) are gaining researcher attention. These are less about structured conversation and more about emergent behavior from many simple agents, often used for simulation and data generation (adjacent to our `fleet-simulator`).

### 3. A Specific, Factual Observation
A clear pattern across commit histories and release notes for CrewAI, AutoGen, and LangGraph in Q1 2026 is the **addition of explicit "interruption" or "human-in-the-loop" nodes**. Previously, human feedback was often bolted on. Now, it's a first-class primitive in the agent graph. For example, LangGraph's documentation now includes a canonical "Human-in-the-Loop" pattern where a node pauses the graph and waits for an external signal (API call, UI input) before proceeding. This suggests a maturation focus: moving from prototypes to operable systems where human oversight is a required feature, not an afterthought.

### 4. Fleet Context Comparison
Our fleet's `holodeck-rust` (sentiment-aware NPCs) and `fleet-simulator` occupy a different niche—simulated environments for training. The mainstream frameworks are primarily tools for **building agent applications**, while our stack includes tools for **training the agents themselves** (`plato-torch`, `plato-ensign`). The "Mirror Play" and "Ghost tile" concepts described in our doctrine are not directly analogous to features in the mainstream frameworks, which are more focused on execution than experiential learning.

### 5. Potential P0 / P1 Considerations for the Fleet
*   **P0 (Negative Space)**: Avoid directly replicating the core orchestration features of CrewAI/AutoGen. That space is crowded and not our primary value.
*   **P1 (Safe Channel)**: Our safe channel appears to be the **training data generation and specialized model fine-tuning loop** (`fleet-simulator` -> `plato-torch` -> `plato-ensign`). The mainstream frameworks' need for better, cheaper, more diverse training simulations could be an opportunity.
*   **P2 (Optimization)**: If in the P1 channel, we could optimize by ensuring our `holodeck-rust` MUD or `fleet-simulator` can export data in a format easily consumed by these popular frameworks for their own agent training.

**Conclusion:** The open-source agent framework ecosystem is in a refinement phase, prioritizing production readiness and control. The fleet's adjacent focus on simulation and embedded training (`cudaclaw`, `flux-runtime`) remains a distinct and potentially complementary track.
