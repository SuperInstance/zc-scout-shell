## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Evolution & Consolidation
The landscape shows a move from monolithic "orchestrator" frameworks toward modular, composable systems. Key themes:
*   **Graph-Based Execution:** Adoption of explicit state graphs (inspired by LangGraph) is now common. Newer frameworks like **Cline** and **Aider** (though primarily code assistants) emphasize deterministic, debuggable workflows over pure LLM chaining.
*   **Rust for Core Runtimes:** Performance-critical components (agent kernels, tool servers) are increasingly written in Rust for memory safety and low-latency interop. The **flux-runtime** project (self-assembling bytecode VM) is an example of this trend.
*   **GPU-Resident Agents:** Experimental projects like **cudaclaw** (noted in fleet context) aim to keep agent state and model weights pinned in GPU memory to avoid PCIe transfer latency, targeting high-frequency trading and real-time simulation.

### 2. What's Hot / Emerging Patterns (P1 Channels)
*   **Constraint-Driven Action:** Frameworks are incorporating hard constraints (budget, safety rules, API quotas) and geometric snapping logic (see **constraint-theory-core**) to reduce hallucinated actions. This aligns with the fleet's P0 (negative space) doctrine.
*   **Self-Training & Specialization:** The pattern of agents generating their own training data (synthetic conversations, code problems) to fine-tune smaller, specialist models is widespread. Projects like **plato-ensign**, which export room experience to LoRA adapters, fit here.
*   **Multi-Agent Simulation as Training Data:** Using simulated environments (e.g., **fleet-simulator**, **holodeck-rust** with sentiment-aware NPCs) to generate structured interaction logs for training is a active research area. It's a scalable source of "tiles."
*   **Local-First & Edge Agents:** With smaller, capable models (DeepSeek-V3, Qwen2.5-7B), there's a push for agents that run entirely on consumer laptops or edge devices (Jetson), reducing cloud dependency and cost.

### 3. Fleet Context Alignment Check
*   **P0 (Negative Space):** The community is recognizing that purely prompt-based orchestration is brittle. Failed agents' lessons ("ghost tiles") are being archived. The deadband protocol's first principle (map what *not* to do) is reflected in this shift toward constraint and validation layers.
*   **P1 (Safe Channels):** The move to graph-based execution, Rust runtimes, and local specialization defines the current "safe channels" for building reliable agents. Our fleet's technologies (**plato-torch**, **cudaclaw**, **constraint-theory-core**) are positioned within these channels.
*   **P2 (Optimization):** Within the graph/channel paradigm, optimization focuses on reducing token use, faster tool selection, and better memory management (vector DBs, incremental summarization).

### 4. Specific, Factual Observation
A notable trend in April 2026 is the **convergence of agent frameworks with game/simulation engines**. Projects are using modified game engines (Bevy/Godot) or MUD servers (**holodeck-rust**) not just for "AI NPCs," but as a controlled, observable environment to train and benchmark multi-agent systems. The simulation produces structured logs (who did what, outcome, sentiment) that are directly consumed as training tiles for fine-tuning. This creates a closed-loop where agents improve by practicing in a synthetic world.

**Recommendation for Fleet:** Continue developing **fleet-simulator** and **holodeck-rust** as primary tile generators. Consider open-sourcing the simulation-to-tile pipeline as a standalone tool; it aligns with the observed market need for high-quality, diverse agent interaction data.
