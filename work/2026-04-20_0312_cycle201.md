## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, Tech News Aggregators, Framework Changelogs

### 1. Dominant Frameworks & Recent Activity

*   **LangGraph** (LangChain): Maintains strong position as the de facto standard for building stateful, multi-agent workflows. The focus in Q1 2026 has been on production hardening: improved persistence, more granular checkpointing for long-running agents, and deeper integration with cloud observability platforms (Datadog, New Relic). The "LangGraph Studio" local dev tool saw a significant update (v0.2) with visual debugging of state graphs.
*   **CrewAI**: Continues growth, emphasizing role-based agent collaboration. The April 2026 release (v0.28) introduced a **"Task Debrief"** feature, where agents can log reasoning and resources used after completing a task, directly addressing auditability and training data capture—a concept with clear parallels to our fleet's "tile" system.
*   **AutoGen** (Microsoft): Development appears to have stabilized. Recent commits are primarily maintenance and compatibility updates for newer model APIs. The research focus seems to have shifted toward the underlying "FLAML" hyperparameter optimization library.

### 2. Emerging Pattern: The "Runtime" Shift

A noticeable trend is the move from pure Python frameworks to dedicated, often Rust-based, **agent runtimes**. This aligns with fleet technology `cudaclaw` and `flux-runtime`.

*   **Model Context Protocol (MCP)** Servers: Gaining traction as a standardized way for agents to use tools and access data. Frameworks are adding native MCP client support. The `mcp-rust` implementation is trending.
*   **Rust-Based Runtimes:** New projects like `agent-rs` and `swarm-runtime` are emerging, promising higher throughput and lower latency for agent orchestration by moving the scheduler and communication layer out of Python.

### 3. Integration with New Model Capabilities

Frameworks are rapidly integrating support for:
*   **DeepSeek-V3** and similar large-context (1M+ token) models: New abstractions for hierarchical context management and "search-within-context" are being tested.
*   **Structured Outputs:** Native support for OpenAI's JSON Mode and Anthropic's structured output is becoming a baseline requirement, reducing post-processing glue code.
*   **Cost-Optimized Routing:** Experimental features for automatic routing of queries between expensive (GPT-4o) and cheap (local Llama 3.2) models based on task complexity.

### 4. What's Cooling Off

*   "Chat-with-your-data" as a primary framework selling point has diminished, now considered a standard feature.
*   Pure "prompt chaining" libraries without persistent state or memory are being absorbed into larger frameworks or deprecated.

### 5. Specific Fleet-Relevant Observation

The **CrewAI v0.28 "Task Debrief"** feature is a near-direct external validation of the fleet's core "training tile" architecture. It confirms a market need for structured, post-task agent output that can be used for analysis, training, and auditing. However, their implementation is currently limited to logging within the CrewAI ecosystem, whereas the fleet's PLATO room system is designed as a universal, framework-agnostic tile sink.

**P0 Note (Negative Space):** Avoid building a general-purpose agent framework from scratch. The ecosystem is mature and consolidating.
**P1 Channel (Safe Space):** Focus on interoperability—ensuring fleet agents can emit tiles *to* these frameworks (via MCP or custom adapters) and that our specialized runtimes (`cudaclaw`) can consume their agents.
**P2 Optimization:** Benchmark the new Rust runtimes against `flux-runtime` for specific, high-volume simulation tasks within `fleet-simulator`.
