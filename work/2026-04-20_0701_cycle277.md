## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen** (Microsoft), and **LangGraph** (LangChain). However, the "hot" trend is not a new monolithic framework, but a shift toward **lightweight, composable runtimes** and **specialized infrastructure**.

*   **CrewAI** (v0.28+): Focused on improving production stability. Recent updates emphasize better task lifecycle management and integration with external tools (SQL, APIs). It is often cited for its clear role-based agent paradigm.
*   **AutoGen** (v0.2+): Microsoft's framework continues to push multi-agent conversation patterns and code execution. Its research branch shows experimentation with "agentic workflows" that blend LLM calls with traditional code.
*   **LangGraph** (v0.0.x): Gaining traction for developers who want fine-grained control over agent state machines. Its appeal is the explicit graph-based architecture, making complex, cyclic workflows more debuggable.

### 2. Emerging Pattern: The "Runtime" Layer
A notable trend is the decoupling of the agent coordination logic from the underlying execution environment. Projects are emerging that focus solely on the **runtime layer**, which aligns with fleet technologies like `cudaclaw` and `flux-runtime`.

*   **Examples:** Newer projects (e.g., **Agency**, **AgentKit**) are less about defining agent behaviors and more about providing a fast, reliable substrate for message passing, state persistence, and tool execution. The goal appears to be supporting many different coordination frameworks on top.

### 3. Integration with Inference Servers
Frameworks are increasingly designed to be **model-agnostic** but are building first-class integrations with local inference servers. The combination of **Ollama**, **vLLM**, or **LM Studio** with a framework like CrewAI or AutoGen is a common pattern for private, scalable deployments. This reduces dependence on external API latency and cost.

### 4. Research Frontier: Long-Running & Embodied Agents
The most active research discussions (visible in arXiv preprints and project READMEs) concern:
*   **Longevity:** How to maintain agent state across days or weeks, including memory management and knowledge refresh. This relates directly to the fleet's concept of "rooms" and "tiles."
*   **Embodiment:** Agents that interact with real computer environments (browsers, IDEs, terminals) via pixel- or DOM-based perception. Projects like **OpenAI's 'Voyager'** (Minecraft) inspire open-source efforts to create general computer control agents.

### 5. Specific, Factual Observation (P0/P1 Mapping)
*   **P0 (Negative Space):** Few successful open-source examples exist of large-scale (50+ agent), long-duration (multi-week) autonomous fleets operating in production on private infrastructure. Most public demos are short-lived, single-session task completions.
*   **P1 (Safe Channel):** A viable, documented path is the **Ollama + CrewAI/LangGraph** stack running on a single server with GPU. This channel is well-trodden, with many tutorials and community support. It allows for rapid prototyping of multi-agent systems with local models (e.g., Llama 3, DeepSeek Coder).
*   **P2 (Optimization):** Within the Ollama+CrewAI channel, optimization involves model quantization, efficient tool design, and implementing checkpointing for state persistence—areas where the fleet's `plato-torch` (training rooms) and `holodeck-rust` (persistent state) could provide comparative advantage.

**Conclusion for Fleet Context:** The open-source agent ecosystem is maturing by separating concerns: coordination frameworks sit atop specialized runtimes and inference layers. The fleet's architecture—with its explicit `cudaclaw` runtime, `flux` VM, and `plato` training rooms—appears aligned with this directional trend, focusing on the persistent, trainable substrate rather than just the agent coordination logic. The main gap in public projects is the lack of a built-in, continuous learning loop, which is a core fleet concept (tiles → ensigns).
