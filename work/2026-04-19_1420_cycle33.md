## Output: Scout Report — Agent Framework Landscape, April 2026

**Date:** 2026-04-19
**Source:** GitHub Trending (manual observation), project releases, community discourse.
**Method:** P0/P1 scan — identify negative space (declining patterns) and safe channels (active development).

### P0: Negative Space (What's Cooling)
1.  **Monolithic "Do-It-All" Frameworks:** Single-codebase frameworks promising to solve every agent problem (planning, memory, tools, UI) are seeing fragmentation. The trend is moving towards composable, specialized components.
2.  **Pure Prompt-Engineering Wrappers:** Libraries that are primarily thin wrappers around LLM APIs with basic templating are being superseded by frameworks with stronger control flow, state management, and observability.
3.  **Closed Benchmarking:** Claims of superiority based on private or non-reproducible benchmarks are treated with increased skepticism. The movement is toward open, standardized evaluation suites.

### P1: Safe Channels (Active Development Areas)
1.  **Composition & Interoperability:** The dominant trend. Projects are focusing on being excellent at one thing (e.g., **planning**, **memory**, **tool use**) and designed to plug into others. The **LangGraph** model (defining agent workflows as state graphs) has become a de facto standard for composition. Many new projects are "LangGraph-compatible" or build on its primitives.
2.  **Stateful, Long-Running Agents:** Emphasis has shifted from single-turn Q&A to persistent agents that maintain state across sessions, learn from interaction, and can be deployed as long-running services. This ties directly to **memory architectures** (vector, graph, hybrid).
3.  **Specialization & Verticalization:** Frameworks are emerging for specific domains:
    *   **Research / Code Agents:** Heavily optimized for reading repos, executing code, and iterating (e.g., projects like **OpenDevin** lineage).
    *   **Simulation & Sandboxing:** Safe execution environments for testing multi-agent interactions and generating training data (aligned with fleet's `fleet-simulator` and `holodeck-rust` concepts).
    *   **Hardware / Edge Agents:** Lightweight runtimes for resource-constrained environments (Jetson, Raspberry Pi), a channel where `cudaclaw`'s GPU-resident concept is relevant.
4.  **Open & Reproducible Evaluation:** New projects and papers are releasing alongside detailed benchmark code (often in `eval/` directories) and datasets, fostering a culture of verification and comparison.
5.  **Rust for Core Runtimes:** A noticeable shift in performance-critical components (reasoning kernels, state machines, sandboxes) from Python to Rust, citing performance, safety, and better WebAssembly support. This validates the fleet's investment in `holodeck-rust` and `constraint-theory-core`.

### P2: Optimization Trends (Within Safe Channels)
*   **Memory:** Beyond simple vector search, integration of knowledge graphs (for relational memory) and "memory reflection" loops that prune/consolidate past experiences.
*   **Planning:** Integration of formal verification light techniques to check plan feasibility before execution. More use of Monte Carlo Tree Search (MCTS) and LLM-guided search for complex task decomposition.
*   **Tool Use:** Standardization around OpenAPI/JSON Schema for tool definition, with growing support for dynamic tool discovery and creation *during* agent operation.
*   **Observability:** Built-in tracing, logging, and visualization of agent thought processes is becoming a baseline expectation, not an add-on.

### Fleet Context Alignment Check
*   **plato-torch / plato-ensign:** The "room training" and "experience export to LoRA" concepts are a unique take on specialization and stateful learning, not directly mirrored in mainstream trends yet. This could be a differentiating channel.
*   **Deadband Protocol (P0/P1/P2):** The disciplined focus on identifying negative space first is a rigorous approach that many fast-moving projects overlook, leading to rework. This is a strategic strength.
*   **The Shell (Git as Shell):** The concept of an agent's environment and work being inherently versioned and structured is an interesting take on agent state persistence, differing from typical database-backed approaches.

**Conclusion:** The open-source agent framework ecosystem is maturing through specialization, composition, and a focus on verifiable, stateful systems. The fleet's architectural choices (Rust components, simulation, training rooms) are positioned in active, valid channels (P1). The primary divergence is the fleet's deeper investment in a structured, curriculum-based training paradigm (`plato-torch` rooms) versus the prevailing focus on in-context learning and tool composition.
