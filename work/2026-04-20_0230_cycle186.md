## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph** (LangChain). A noticeable trend is the shift from pure "orchestration" to **specialized, lightweight runtimes** that can be embedded closer to hardware or within larger systems. The "GPU-resident agent" concept, similar to fleet's `cudaclaw`, is gaining traction in research repositories.

### 2. What's New/Hot (April 2026)
*   **Model Roulette & Multi-Model Routing:** Frameworks are increasingly building native support for dynamically routing queries between multiple LLM providers (OpenAI, Anthropic, Google, Groq, local models) based on cost, latency, and task type. This is often paired with fallback mechanisms.
*   **Rust-Based Runtimes:** There is growing interest in Rust for building the core runtime of agent systems, emphasizing safety, performance, and small binary sizes. Projects like `holodeck-rust` in our fleet align with this trend.
*   **"Self-Improving" & "Self-Training" Loops:** Inspired by academic papers on LLM self-improvement, several new experimental frameworks are implementing closed-loop systems where agents generate their own training data, critique their outputs, and fine-tune small adapters (LoRA). This mirrors the `plato-torch` and `plato-ensign` concept.
*   **Local-First & Edge Agents:** With the release of more powerful small language models (SLMs) capable of running on consumer hardware, there's increased development activity around agents that operate fully offline or on edge devices (Jetson, Raspberry Pi). Optimization for Ollama and LM Studio is common.

### 3. Specific, Factual Observation: The "Validation Layer"
A pattern emerging in the most active repositories (e.g., `ai-agent-benchmark`, `agent-eval`) is the formalization of an **agent validation layer**. This is no longer just unit tests. It involves:
*   **Scenario-Based Testing:** Running agents through complex, multi-step user scenarios and scoring final outcomes.
*   **Cost & Latency Tracking:** Automated auditing of API calls and token usage per task.
*   **Deterministic Testing:** Using constrained, deterministic "sandbox" environments (like simple MUDs or game engines) to test an agent's reasoning and instruction-following without API variability.

This suggests the community's focus is maturing from "getting agents to run" to "ensuring agents are reliable, cost-effective, and predictable in production-like conditions." The fleet's `fleet-simulator` and the concept of "training tiles" submitted for validation fit within this broader industry direction.

### 4. Fleet Contextual Note
The Cocapn fleet's architecture—particularly the separation of **training rooms** (`plato-torch`), **experience export** (`plato-ensign`), and **specialized runtimes** (`cudaclaw`, `holodeck-rust`)—appears ahead of the curve in its modular, research-oriented design. The mainstream open-source frameworks are typically more monolithic. The fleet's "Deadband Protocol" (P0/P1/P2) is a unique contribution to agent safety/planning not widely seen elsewhere.
