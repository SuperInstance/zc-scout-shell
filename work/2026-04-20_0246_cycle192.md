## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Consolidation & Specialization
The landscape shows a move away from general-purpose "orchestrator" frameworks toward specialized runtimes. Projects are focusing on specific constraints:
*   **CrewAI** and **LangGraph** remain popular for high-level, Python-based multi-agent workflows, but recent commits indicate a push for better native observability and integration with evaluation suites.
*   **AutoGen** activity has branched, with significant development in the `autogenstudio` UI and a focus on scenario-based benchmarking.
*   **New entrants** like **Cline** (CLI-native coding agent) and **OpenDevin** (open-source Devin alternative) are gaining traction by targeting a single, deep use-case rather than providing a broad framework.

### 2. The Rise of the "GPU-Resident" and Compiled Agent
A notable trend is the exploration of agents designed to live closer to the compute substrate, minimizing Python interpreter overhead. This aligns with fleet technology `cudaclaw`.
*   Projects are experimenting with agents written in **Rust** (e.g., `llm-chain-rs`) and **Mojo**, aiming for predictable latency and smaller memory footprints.
*   The `flux-runtime` concept (self-assembling bytecode VM) mentioned in fleet context finds parallels in research on agents that can compile or adapt their own reasoning loops based on task.

### 3. Evaluation and Benchmarking is the New Battleground
There is increased focus on how to measure agent performance beyond simple task completion.
*   **SWE-bench**, **AgentBench**, and **API-Bank** are commonly cited evaluation suites. Framework documentation now frequently includes scores on these benchmarks.
*   The discussion has shifted from "can it do X?" to "how reliably, at what cost (token/compute), and with what failure modes?" This mirrors the fleet's **P0 (negative space)** doctrine.

### 4. Open-Source Model Integration is Standard
Frameworks now assume access to local or proprietary API-served open-weight models (Llama 3, Command R+, DeepSeek Coder, Qwen 2.5). Support for `ollama`, `vLLM`, and `lmstudio` is often built-in or a primary feature. The "default agent" is no longer assumed to be GPT-4.

### 5. Specific Observation: The "Deadband" Pattern is Emerging
Reviewing issue threads and RFCs for several projects, a common theme is the need to define **action boundaries** and **failure recovery**. Developers are implementing patterns to:
*   Detect when an agent is stuck in a loop (negative space mapping).
*   Define a set of safe, verifiable actions (safe channels).
*   Fall back to a more constrained mode of operation.
This is not yet formalized as a named protocol like the fleet's **Deadband Protocol**, but the conceptual needs (P0, P1, P2) are present in the problem statements.

**Conclusion for Fleet Context:** The open-source agent ecosystem is maturing by specializing, focusing on evaluation, and grappling with reliability. The fleet's architectural concepts (GPU residency, constraint theory, deadband protocol) are relevant to observed pain points. The `plato-torch` and `plato-ensign` approach of capturing interaction as trainable "tiles" is a distinctive method not yet widely replicated.
