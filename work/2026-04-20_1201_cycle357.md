## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph**. A noticeable trend is the consolidation of features rather than the emergence of entirely new frameworks. Projects are focusing on **production readiness**, **observability**, and **cost control**.

*   **CrewAI (v0.28+):** Emphasis has shifted towards **orchestration** and **task management**. Recent updates include improved support for long-running workflows and better integration with external tooling (e.g., SQL databases, custom APIs). The community is building "crews" for specific verticals (e.g., customer support, code review).
*   **AutoGen (Microsoft):** Development continues to focus on **multi-agent conversations** and **research applications**. A significant area of activity is around **retrieval-augmented generation (RAG)** integrations and optimizing agent-to-agent communication patterns to reduce token usage.
*   **LangGraph (LangChain):** Gaining traction for developers who want **fine-grained control** over agent state and execution flow. Its graph-based paradigm is being used to build complex, cyclic workflows that are easier to debug than linear chains. The `StateGraph` abstraction is a key differentiator.

### 2. Emerging Pattern: The "Runtime" Layer
A new layer is emerging **beneath** the framework level, focused on **execution efficiency** and **hardware utilization**. This aligns with fleet technologies like `cudaclaw` and `flux-runtime`.
*   Projects like **`inference`** (from the `unsloth` team) and **`vLLM`** are being adapted not just for model serving, but as backbones for agent runtimes, emphasizing high-throughput, low-latency execution.
*   There is research interest in **GPU-resident agents**, where agent state and logic persist in GPU memory to avoid costly CPU-GPU transfers, mirroring the `cudaclaw` concept.

### 3. Integration with "Smaller" Models
There is a measurable shift towards using **smaller, fine-tuned models** (7B-13B parameters) for specific agent tasks, paired with larger models for planning or review. This is driven by cost and latency concerns. The `plato-ensign` project's concept of exporting room experience as LoRA adapters fits directly into this trend.

### 4. What's Not Hot (P0: Negative Space)
*   **Monolithic, all-in-one frameworks** claiming to solve "general AI." The market is skeptical.
*   **Proprietary, closed-agent systems** with high per-task pricing are facing pushback in favor of open, customizable stacks.
*   **Agents without transparent reasoning** or explainability features are harder to deploy in production environments.

### 5. Specific, Factual Observation for the Fleet
**Observation:** The open-source agent ecosystem is bifurcating. The **application layer** (CrewAI, AutoGen) is stabilizing around workflow design. The **innovation battleground** has moved to the **systems layer**: efficient runtime execution (GPU residency, bytecode VMs like `flux-runtime`), cost-effective fine-tuning for specialization (like `plato-ensign`), and simulation environments for training (like `fleet-simulator`). Projects that combine these systems-level innovations with robust orchestration appear to have traction.

**Recommendation (P1/P2):** The fleet's investment in `cudaclaw`, `flux-runtime`, and `plato-ensign` is aligned with the observed vector of innovation. A potential safe channel (P1) is to document and demonstrate how these systems-layer components can be integrated with a mainstream framework like LangGraph to create a high-performance, trainable agent stack.
