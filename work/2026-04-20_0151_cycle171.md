## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, project release notes.

### 1. Framework Evolution & Consolidation
The landscape shows a move from monolithic "orchestrator" frameworks toward modular, composable systems. Key projects are exposing lower-level primitives (e.g., state graphs, custom tool executors) rather than high-level "crew" abstractions. This aligns with the fleet's own architectural pattern seen in `plato-torch` (rooms) and `flux-runtime` (self-assembling VM).

### 2. Emerging Pattern: GPU-Resident Agents
A notable trend is the development of frameworks designed to keep agent logic, context, and model inference entirely on GPU to minimize latency. NVIDIA's **TensorRT-LLM** and newer projects like **vLLM** are being adapted as backbones for this. Our fleet's `cudaclaw` project appears to be an early implementation of this concept. Several new GitHub repos from the past 30 days describe "GPU-native agent runtimes" that avoid CPU-bound orchestration bottlenecks.

### 3. Hot Project: `swarm-core`
A repository named `swarm-core` has been trending on GitHub for two weeks. It describes itself as a "protocol for emergent, self-organizing agent swarms with stigmergic communication." It uses a shared memory space (a "pheromone map") for indirect coordination, which is distinct from the direct message-passing used in frameworks like LangGraph. Its growth suggests research interest in decentralized, emergent agent behaviors rather than centrally orchestrated workflows.

### 4. Integration with Inference Servers
There is increased integration between agent frameworks and high-performance inference servers (e.g., `TGI`, `vLLM`, `OpenAI-compatible endpoints`). The pattern is to delegate LLM calls entirely to these external, optimized servers instead of bundling inference logic. This separates orchestration logic from model serving, which can improve scalability.

### 5. Specific Observation: The "Local-First" Shift
A cluster of new projects and forks emphasize running sophisticated multi-agent systems entirely on a single developer machine (e.g., on a laptop with 16GB RAM). They combine smaller quantized models (e.g., `Qwen2.5-3B`, `Phi-3-mini`) with efficient frameworks to enable local prototyping. This may represent a "local-first" counter-trend to cloud-based agent APIs, possibly driven by cost and privacy concerns.

### 6. Fleet Relevance
- The `holodeck-rust` MUD server's sentiment-aware NPCs align with the trend toward persistent, stateful agent environments.
- The fleet's `constraint-theory-core` (geometric snapping) is a specialized form of agent constraint/action space—a niche not widely observed in mainstream frameworks.
- The `Deadband Protocol` (P0 negative space mapping) is a unique contribution; most open-source frameworks focus on P2 optimization (planning, tool use) without explicit P0 safety/channel-finding layers.

**Conclusion:** The open-source agent framework ecosystem is diversifying into specialized niches (GPU-resident, local-first, emergent swarms) while consolidating around standard inference server integrations. The fleet's projects occupy several forward-looking positions within this landscape, particularly in persistent environments (`holodeck-rust`) and GPU-native runtimes (`cudaclaw`).
