## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, release notes.

### 1. Framework Consolidation & Specialization
The landscape shows a move away from general-purpose "orchestrator" frameworks toward specialized runtimes. Key trends:
*   **CrewAI** and **LangGraph** remain popular for prototyping multi-agent workflows with clear state machines.
*   **AutoGen** is seeing increased use in academic and research settings for its conversational patterns, but some users report complexity in production deployment.
*   **New entrants** like **Cline** (code-centric) and **OpenDevin** (open-source Devin alternative) are gaining traction by focusing on a single, deep use case (software development).

### 2. The "Runtime" Shift
A notable pattern is the emergence of frameworks designed as **long-lived, stateful services** rather than script orchestrators. This aligns with the fleet's `cudaclaw` and `flux-runtime` concepts.
*   **Dify** and **FastGPT** are trending as low-code platforms that bundle models, tools, and a persistent backend.
*   Projects emphasizing **GPU-residency** (keeping agent state in VRAM) and **WebAssembly sandboxing** for tool execution are receiving increased attention from performance-focused developers.

### 3. Integration with Emerging Models
Frameworks are rapidly adding support for new model APIs.
*   **DeepSeek-V3** and **DeepSeek-R1** integrations are common, given the model's strong reasoning and open weights.
*   **Groq** (LPU) support is often highlighted for latency-sensitive applications.
*   There is growing experimentation with **small, fine-tuned models** (7B-13B parameter) as specialized "worker" agents within a larger system, a pattern similar to the fleet's "ensign" concept.

### 4. What's NOT Hot (P0: Negative Space)
*   **Monolithic, do-everything frameworks** are being criticized for complexity.
*   **Pure Python orchestrators** with high latency between agent steps are being questioned for production use.
*   **Projects without a clear deployment story** (Docker, Kubernetes) are losing visibility.

### 5. Specific Fleet-Relevant Observation
The **plato-torch** and **plato-ensign** approach of "training rooms" that output portable adapters (LoRA) is a distinct pattern. Most open-source agent training focuses on:
1.  **Reinforcement Learning from Human Feedback (RLHF)** for single models.
2.  **Curating datasets** of interaction traces.
The fleet's method of accumulating "tiles" from operational work to create specialized "ensigns" represents a different, potentially complementary, path focused on **continuous, incremental specialization** from multi-agent interaction data.

**Recommendation for Fleet Context:** The trend toward specialized, stateful runtimes validates architectural directions like `cudaclaw`. The fleet's "room-to-ensign" training pipeline remains a unique differentiator. Monitoring projects that treat agent interactions as a training dataset (e.g., **OpenAI's "Agent School"** research) could provide useful P1 channels for methodology.
