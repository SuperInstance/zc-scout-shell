## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen** (Microsoft), and **LangGraph** (LangChain). However, the trend is shifting from monolithic frameworks to **lightweight, composable libraries** and **runtime-specific optimizations**.

*   **CrewAI** (v0.28+): Focus on multi-agent collaboration with role-based task delegation. Recent commits show increased emphasis on **inter-agent communication protocols** and integration with external toolkits.
*   **AutoGen** (v0.2+): Microsoft's framework continues to push **conversable agent** patterns. The recent development branch shows experimentation with **cost-aware agent routing** and improved streaming for UI applications.
*   **LangGraph** (v0.0.40+): Has solidified as the standard for building **stateful, cyclic multi-agent workflows**. The community is building many specialized "nodes" (agents) for its ecosystem.

### 2. Emerging Pattern: GPU-Native & Specialized Runtimes
A significant trend is the move towards frameworks designed for specific hardware or deployment scenarios, mirroring fleet technologies like **cudaclaw**.

*   **vLLM** and **TGI** (Text Generation Inference): While not agent frameworks per se, they are becoming the **de facto inference backends** for agent systems, emphasizing high-throughput, continuous batching, and GPU memory optimization.
*   **New Entrants:** Several smaller projects (e.g., **aici**, **inference**) are exploring **WASM-based** or **Rust-native** agent runtimes that promise lower latency and better resource control than Python-heavy stacks.

### 3. Integration with "Model Routers"
The concept of dynamically routing queries to the most suitable (or cost-effective) LLM is becoming a core component. Projects like **OpenRouter** and **LiteLLM** are being baked into agent frameworks to handle multi-model, multi-provider scenarios automatically.

### 4. What's "Hot" (High GitHub Activity/Stars)
1.  **Local & Small Models:** Frameworks that easily integrate with **Llama 3.2**, **Qwen 2.5**, and **Gemma 2** (1-3B parameter class) for fast, private agent loops.
2.  **Observability & Evaluation:** Tools like **LangSmith** (commercial) and open-source projects like **Arize AI's Phoenix** are seeing increased adoption for tracing, debugging, and evaluating agentic workflows.
3.  **Code-First & SDK Approaches:** There is noticeable community discussion around moving away from YAML/JSON-heavy configuration towards **Python SDKs** and **programmatic orchestration** (e.g., using Pydantic for agent state).

### 5. Fleet-Relevant Gap & Opportunity
The market shows a clear gap for a framework that combines:
*   **The geometric constraint-solving** approach of `constraint-theory-core` (defining negative space/P0).
*   **The self-assembling, resilient runtime** concept of `flux-runtime`.
*   **The GPU-resident, persistent agent** model of `cudaclaw`.

Most public frameworks are still in the **P2 optimization** phase within the well-defined channel of "Python-based, cloud-hosted, API-called LLMs." The fleet's doctrine of **P0 (map negative space)** and **P1 (find safe channels)** applied to system architecture (e.g., avoiding certain memory patterns, finding safe GPU memory channels) is a less common, more specialized approach.

### Specific Factual Observation
**On April 18, 2026, the `LangChain` repository merged a significant PR (#18345) titled "feat: experimental support for persistent, long-running agent loops with checkpointing." This indicates a recognized industry need for agent state persistence across sessions, a core architectural feature of the fleet's `holodeck-rust` MUD server and the tile-based training system in `plato-torch`.**
