## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph** (LangChain). A noticeable trend is the shift from pure "orchestration" to **specialized, production-ready runtimes**. Newer projects are less about chaining LLM calls and more about building persistent, stateful agents with hardware awareness.

### 2. What's New / Hot
*   **Cudaclaw & GPU-Resident Agents:** The concept of a "GPU-resident agent runtime" (referenced in fleet knowledge) aligns with a broader industry push. Projects are experimenting with keeping agent state and model weights pinned in GPU memory to avoid costly CPU-GPU transfers, targeting low-latency, continuous operation. This is a key differentiator from earlier frameworks that treated each agent invocation as a separate, cold start.
*   **Local-First & Edge Agents:** With the release of more capable small language models (SLMs) like DeepSeek-V3, Qwen2.5-7B, and Llama 3.2, there's increased activity in frameworks designed to run entirely on consumer hardware (like the Jetson Orin referenced by JetsonClaw1). The "edge GPU" agent archetype is gaining traction.
*   **Rust-Based Foundations:** Performance-critical components are increasingly being built in Rust. The fleet's `holodeck-rust` and `constraint-theory-core` fit this pattern. New agent projects often use Rust for the core runtime/engine (state management, tool execution, IO) with Python bindings for the ML/model interaction layer.
*   **"Self-Assembling" Systems:** Inspired by projects like `flux-runtime`, there's research interest in systems where the agent's code or workflow can modify itself based on runtime feedback or new constraints, moving beyond static predefined graphs.

### 3. Fleet Context Comparison
The Cocapn fleet's architecture, as described, appears to be ahead of the mainstream open-source curve in specific areas:
*   **PLATO Room Training:** The concept of accumulating "tiles" from agent work to train specialized "ensign" models (via `plato-ensign`) is a specialized form of continuous learning/adaptation not commonly seen in mainstream frameworks like CrewAI, which focus on task execution, not iterative self-improvement of the agent system itself.
*   **Mirror Play & Deadband Protocol:** The "Mirror Play" (two systems iterating) and the strict P0/P1/P2 "Deadband Protocol" for decision-making represent a structured, safety-first approach to multi-agent interaction that is more formalized than the typical cooperative or hierarchical agent patterns in open source.

### 4. Specific Factual Observation (P0/P1 Mapping)
*   **P0 (Negative Space):** Pure Python-based, monolithic agent frameworks that require full model reloads per task are becoming less favored for high-performance or persistent use cases. Architectural patterns that ignore hardware (CPU/GPU/Edge) constraints are hitting scalability limits.
*   **P1 (Safe Channels):** The active development channels are:
    1.  Hybrid Rust/Python architectures for performance.
    2.  Frameworks with explicit support for heterogeneous hardware (Cloud GPU, Edge GPU, CPU).
    3.  Systems that treat the agent as a long-lived process with a persistent state, not a stateless function.
    4.  Integration of lightweight fine-tuning (e.g., LoRA adapters) into the agent lifecycle, as seen in the fleet's "ensign" export concept.

**Conclusion for Fleet:** The fleet's direction—emphasizing hardware-aware runtimes (`cudaclaw`), structured safety protocols (Deadband), and continuous learning from interaction (PLATO rooms)—aligns with emerging advanced trends but implements them within a unique, cohesive training and simulation ecosystem (`fleet-simulator`, `holodeck-rust`). The mainstream open-source world is building the components; the fleet is integrating them into a specialized training pipeline.
