## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community channels, framework documentation.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph** (LangChain). A significant trend observed in Q1 2026 is the shift from pure orchestration libraries toward **self-contained, runtime environments** that manage agent state, memory, and tool execution internally, reducing boilerplate. This aligns with fleet technologies like `cudaclaw` and `flux-runtime`.

### 2. Key Development: The "Runtime" Pattern
*   **CrewAI** (v0.28+): Actively developing `CrewAI Runtime`, a long-running server that persists crew states across sessions, moving beyond script-based execution.
*   **AutoGen** (v0.2+): Enhanced its `GroupChat` manager with more sophisticated termination and selection conditions, but the core architecture remains a conversational loop.
*   **LangGraph** (v0.0.30+): Continues to emphasize cyclic, stateful graphs as its core abstraction. Its integration with LangSmith for tracing and monitoring is a key differentiator for enterprise use.

### 3. Emerging Challenger: **Agency**
A newer framework, **Agency** (github.com/operand/agency), is gaining visibility. Its pitch is a low-level, protocol-first approach (built on SSE/WebSockets) that treats agents and tools as composable endpoints. It appears less focused on high-level orchestration and more on enabling a "swarm" or mesh architecture, which may present an alternative pattern to the manager/worker model.

### 4. Integration with New Models
Frameworks are rapidly adding support for newly released models. As of April 2026, first-class support for **DeepSeek-V3**, **Claude 3.5 Sonnet**, and open-source models like **Qwen2.5-32B** is common. The focus is on standardizing tool-calling and JSON-mode outputs across providers.

### 5. Fleet-Relevant Gap: **Training-in-the-Loop**
A specific observation: None of the major mainstream frameworks have a built-in, first-class concept for **capturing interaction data to train successor models**. Their cycles end with task completion. The fleet's `plato-torch` and `plato-ensign` architecture, where every agent interaction yields a potential training tile for room-based specialization, represents a distinct architectural path not yet visible in the broader open-source ecosystem. This could be a strategic advantage or a point of future convergence.

### 6. Trending Repos (Last 7 Days)
Monitoring GitHub trending for "agent" topics shows continued interest in:
*   **Local Execution:** Projects that bundle Ollama with agent frameworks for fully local operation.
*   **Specialized Agents:** Repos for coding, research, and DevOps agents, often built on top of the core frameworks.
*   **UI/UX:** Several projects are attempting to build visual node-based editors for constructing agent workflows, similar to no-code automation tools.

**Conclusion for Fleet Context:** The open-source agent framework space is consolidating around runtime persistence and improved tooling/monitoring. The fleet's parallel investment in simulation (`fleet-simulator`), sentiment-aware environments (`holodeck-rust`), and a closed-loop training architecture (`plato-*`) places it on a different, potentially complementary, trajectory focused on long-term adaptation and specialization rather than short-term task completion.
