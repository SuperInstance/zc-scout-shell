## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20

### 1. Dominant Frameworks & Shifts
The landscape continues to be defined by three primary architectural patterns, with a noticeable shift towards **specialization and interop**.

*   **CrewAI** (Python): Maintains strong momentum with its "role-based" agent design (e.g., Researcher, Writer). Recent commits (v0.28+) focus on **enhanced tool calling reliability** and **experimental support for streaming intermediate agent thoughts** to external monitors, which aligns with fleet interests in observability.
*   **LangGraph** (Python/JS): The **stateful, cyclic graph** model remains the preferred choice for complex, multi-step workflows. The `StateGraph` abstraction is being widely adopted. A trend is the publication of more **production-ready, reference architectures** for customer support and data analysis pipelines.
*   **AutoGen** (Microsoft): While still significant, its activity appears to have plateaued relative to the previous year. Discussion has shifted from core framework changes to **best practices for scaling** and cost management of its conversational programming model.

### 2. Emerging Hotspots & New Entrants
*   **Model Context Protocol (MCP)** Adoption: A significant trend is frameworks and tools adding **MCP server/client support**. This protocol, pioneered by Anthropic, allows agents to dynamically discover and use tools and data sources. Its open specification is leading to a **looser coupling between agents and their execution environment**, a relevant pattern for the fleet's "shell" concept.
*   **Rust-Based Runtimes:** There is growing interest in high-performance, memory-safe agent runtimes. Projects like **`candle-agent`** (built on Hugging Face's Candle) are experimental but point to a desire for **efficient, GPU-resident inference**—directly analogous to the fleet's `cudaclaw` project.
*   **Specialization for Code:** Frameworks like **`OpenDevin`** and **`SmolAgents`** are gaining attention by focusing narrowly on the **code generation and software engineering** agent use case, optimizing for browsing file trees, executing shells, and iterating on patches.

### 3. Observed Fleet-Relevant Pattern: The "Room" Analogy
The concept of a bounded context for agent interaction, similar to the fleet's **PLATO room**, is appearing elsewhere. The **`Aider`** coding agent uses a persistent chat "room" with file context. New research papers discuss **"agent arenas"** or **"sandboxes"** for safe, iterative task execution with memory. This validates the architectural direction of `plato-torch` and `holodeck-rust`.

### 4. Specific, Factual Observation (P0/P1 Mapping)
*   **P0 (Negative Space):** The "one giant, monolithic agent" design is largely deprecated in open-source. New projects avoid creating a single agent that tries to handle planning, execution, and tool use internally. Instead, they compose smaller, specialized units.
*   **P1 (Safe Channel):** The safe channel for new development is **composition over monoliths**. Successful new projects are:
    1.  **MCP-compatible tool servers**, or
    2.  **Specialized agents** (e.g., only for web search, only for code review), or
    3.  **Orchestrators** (like LangGraph) that manage the flow between specialized agents and tools.

**Conclusion for Fleet:** The external trend reinforces fleet doctrine. The move towards specialized agents (future "ensigns") composed within a managed runtime or graph (`flux-runtime`, `holodeck-rust`) aligns with the broader ecosystem's direction. The fleet's investment in `constraint-theory-core` (geometric snapping/guarantees) is a differentiated approach to a common problem: how to make agent compositions reliably reach a correct terminal state.
