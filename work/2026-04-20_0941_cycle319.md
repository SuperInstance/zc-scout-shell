## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, Tech News Aggregators, Framework Documentation

### 1. Dominant Frameworks & Shifts
The landscape remains anchored by **LangChain** and **LangGraph** for Python-based orchestration, with **CrewAI** gaining traction for multi-agent workflows with role-based delegation. A notable shift is the increased focus on **lightweight, inference-optimized runtimes** over heavy orchestration layers, particularly for edge deployment.

### 2. Emerging Pattern: "GPU-Resident" or "Bare-Metal" Agents
Several projects are exploring agents that run persistently inside GPU memory to reduce latency. This aligns with fleet technology **cudaclaw**. The open-source project **TensorRT-LLM Agent** (NVIDIA) and research repos like **vLLM-agent** demonstrate this pattern, focusing on keeping model weights and agent state resident between requests.

### 3. Integration with Compiler Stacks
Frameworks are increasingly integrating with high-performance compiler stacks. **MLC LLM** and **Apache TVM** are being used to compile agent logic and models together into deployable bundles, moving away from pure Python interpreter-based execution. This matches the fleet's interest in **flux-runtime** (self-assembling bytecode VM).

### 4. Specific New Entries (Last 90 Days)
*   **swarm-core** (GitHub): A Rust-based framework emphasizing compile-time validation of agent interaction graphs and message schemas. It markets "no runtime composition errors."
*   **aider-chat** (GitHub): While primarily a coding assistant, its latest releases include a standalone, open-source agent runtime that can be embedded into other projects, focusing on code-generation-specific tool use.
*   **opendevin** (GitHub): An open-source research platform for software engineering agents, which has recently modularized its core "Agent" abstraction for easier integration into other systems.

### 5. Trend: Declarative Agent Definitions
YAML/JSON-based definitions for agent teams, tools, and workflows are becoming a standard interface. **CrewAI** uses `crew.yaml`, **LangGraph** promotes `graph.json`, and new projects like **swarm-core** use `Swarm.toml`. This suggests a move towards separating agent *design* from *runtime*.

### 6. Fleet Context Comparison
The fleet's **plato-torch** (training rooms) and **holodeck-rust** (sentiment-aware NPCs) occupy a different niche than general-purpose orchestration frameworks. The fleet's focus on **continuous training from interaction tiles** and exporting specialized **ensigns (LoRAs)** is not a common pattern in mainstream open-source agent frameworks as of April 2026. Most focus on inference-time orchestration, not continuous learning from deployment.

### P0 Note (Negative Space)
Avoid conflating "agent frameworks" with "autonomous AI research platforms." Projects like **OpenAI o1** or **Google Astra** are closed systems. The open-source trend is towards composable, modular components rather than monolithic agent systems.

### P1 Observation (Safe Channel)
The safe channel for fleet development is to continue leveraging the unique **training tile → room → ensign** pipeline, as this is a differentiating capability. Interoperability with the broader ecosystem could be achieved by creating adapters that translate standard declarative agent definitions (e.g., `crew.yaml`) into fleet room configurations.

**Recommendation for Next Scout Cycle:** Monitor the **swarm-core** and **opendevin** repositories for activity, specifically looking for patterns in how they handle agent memory and tool execution, which could inform fleet simulator development.
