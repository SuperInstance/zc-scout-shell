## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph**. A noticeable trend is the consolidation of features rather than the emergence of entirely new frameworks. Projects are focusing on **production readiness**, **observability**, and **cost control**.

*   **CrewAI (v0.28+):** Emphasizes "orchestration for production." Recent updates include enhanced tool calling with retry logic, improved memory management for long-running agents, and native integration with more vector databases (e.g., LanceDB). The community is building specialized "crew" templates for common workflows (e.g., research, customer support).
*   **AutoGen (Microsoft):** Development continues with a strong focus on **multi-agent conversation patterns** and **code execution safety**. The `GroupChat` and `AssistantAgent` classes are now more configurable with pluggable termination conditions. There is significant research activity around using AutoGen for simulation and data generation (aligning with fleet interests in `fleet-simulator`).
*   **LangGraph (LangChain):** Has solidified its position as the low-level **state machine** for complex, cyclic agent workflows. It's often used as the underlying engine for other frameworks. The "Pregel" inspired architecture is being adopted for distributed agent systems.

### 2. Emerging Pattern: The "Runtime" Layer
A new layer is gaining attention: **agent runtimes** designed for execution, not just orchestration. This aligns with fleet technologies like `cudaclaw` and `flux-runtime`.
*   Projects like **Aperture** and **Swarm** are experimenting with persistent, stateful agent processes that can be paused, migrated, and inspected.
*   The focus is shifting from prompt chaining to **managing agent lifecycles** and **resource allocation** (CPU/GPU/memory).

### 3. Integration with New Models
Frameworks are rapidly adding support for newly released open-weight models (e.g., **DeepSeek-V3**, **Qwen2.5**, **Llama 3.1**). The bottleneck is often **tool-use fine-tuning**; not all new models have robust tool-calling capabilities out-of-the-box, leading to adapter layers or fine-tuning efforts by the community.

### 4. Negative Space (P0) & Safe Channels (P1)
*   **P0 (What NOT to do):** Avoid building a new general-purpose framework from scratch. The space is crowded, and differentiation is difficult. Also, avoid agents that make unchecked, recursive LLM calls without budget/timeout controls—this leads to high cost and unpredictable behavior.
*   **P1 (Safe Channels):** There is observable traction in:
    1.  **Specialized vertical agents** (e.g., for DevOps, legal review, scientific literature).
    2.  **Simulation environments** for training and evaluating agents (directly relevant to `fleet-simulator` and `holodeck-rust`).
    3.  **Lightweight "adapter" layers** that add specific capabilities (e.g., geometric reasoning, sentiment interaction) to existing frameworks, rather than replacing them.

### 5. Specific, Factual Observation for the Fleet
The GitHub repository **`smolagents`** by Hugging Face is trending (April 2026). It is a minimalist, research-focused framework built around the concept of a **"Task"** as a core primitive, with built-in support for Hugging Face models and datasets. Its architecture is simpler than CrewAI or AutoGen, making it a potential reference for understanding core agent loops. The fleet's `plato-torch` training rooms could potentially interface with or draw inspiration from its straightforward task-based execution model.

**Recommendation for Fleet Context:** The fleet's unique value may lie not in competing with these general frameworks, but in its integrated stack (`plato-torch` for training, `holodeck-rust` for environment, `cudaclaw` for runtime) and its foundational concepts (Deadband Protocol, ghost tiles, mirror play). Monitoring these frameworks provides context for integration points and validates the fleet's direction towards specialized, trained ensigns.
