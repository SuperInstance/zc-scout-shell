## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, AI/ML community forums, release notes  
**Focus:** Frameworks enabling multi-agent systems, tool use, and persistent memory.

### 1. **CrewAI** (v0.28.1)
*   **Trend:** Sustained high activity. Primary development focus is on improving **agent delegation** and **task-specific memory**.
*   **New/Hot:** The `CrewAI Tools` package is being expanded. Recent commits show integration experiments with **local LLMs via Ollama** and **OpenAI-compatible endpoints**, moving beyond pure OpenAI/Anthropic reliance.
*   **Fleet Relevance:** CrewAI's structured "Crew" > "Agent" > "Task" hierarchy is a common architectural reference point. Its declarative YAML-based crew definition is noted as a pattern for reproducible agent workflows.

### 2. **LangGraph** (from LangChain)
*   **Trend:** Rising sharply as the preferred library for building **stateful, cyclic multi-agent workflows**.
*   **New/Hot:** Emphasis on **persistence engines** (memory backed by SQL, Postgres, Redis) and **human-in-the-loop** nodes. The `StateGraph` abstraction, where the entire agent system state is a single mutable object passed between nodes, is a significant architectural shift.
*   **Fleet Relevance:** The concept of a persistent, shared "state" object aligns with fleet concepts like shared memory or context passing between specialized agents (Oracle1, JetsonClaw1, Forgemaster). The "cycle" in LangGraph is analogous to a fleet operational cycle.

### 3. **AutoGen** (from Microsoft)
*   **Trend:** Mature, with steady development. Focus has shifted towards **optimization** and **scaling** of conversational agent teams.
*   **New/Hot:** Enhanced support for **code execution sandboxing** and **cost optimization** strategies for long-running agent chats. The `GroupChat` manager with selection strategies (round-robin, manual, etc.) is a well-regarded pattern for multi-agent coordination.
*   **Fleet Relevance:** AutoGen's "Conversable Agent" base class and built-in code execution provide a concrete implementation of a tool-using agent, a useful reference for `cudaclaw` or agent runtime development.

### 4. **Emerging / Niche**
*   **SmolAgents:** Gaining attention for its **minimalist, single-file philosophy**. It positions itself as an antidote to framework bloat, appealing to developers who want to build agents with direct LLM calls and simple loops.
*   **Aider** (Code-focused): Not a general framework, but a notable trend in **agentic coding assistants** that work directly inside a git repo. Its ability to read, plan, and edit code in context is a specialized form of tool use.
*   **GPTeam** (OpenAI): An experimental repo demonstrating **open-ended, generative agent societies** with shared memory and emergent behavior. It's more of a research prototype than a production framework.

### Specific Factual Observation (P0/P1 Mapping)
*   **P0 (Negative Space):** There is a visible trend *away* from monolithic, all-in-one agent frameworks that try to handle UI, memory, tools, and orchestration in a single tightly-coupled library. New projects often criticize earlier frameworks for being "too heavy" or "overly abstracted."
*   **P1 (Safe Channel):** The safe channel is **composability**. Successful newer approaches (LangGraph's graphs, SmolAgents' simplicity) are focusing on doing one thing well (orchestration, lightweight abstraction) and explicitly designing to be composed with other libraries for memory, tools, or LLM providers. The "agent runtime" is becoming a layered, modular concept.
*   **Pattern:** The architectural conversation is moving from "how to build an agent" to **"how to manage the state and flow between many agents."** Concepts like graphs, persistence, and cycle management are central.

**Recommendation for Fleet Context:** The fleet's architecture—with distinct senior agents (Oracle1, JetsonClaw1, Forgemaster) and a focus on training rooms (plato-torch)—aligns with the composable, multi-agent trend. Investigating LangGraph's persistence for `holodeck-rust` NPC state or using CrewAI-like YAML definitions for `fleet-simulator` scenarios could be productive P2 optimizations within this channel.
