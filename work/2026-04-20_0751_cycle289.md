## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20

### 1. Framework Landscape & Movement
The dominant pattern is **multi-agent specialization** and **runtime unification**. Single, monolithic frameworks are giving way to modular systems where different agents handle specific roles (like our fleet's Navigator, Sentinel, Scribe). There is a noticeable push towards frameworks that can deploy across heterogeneous hardware (cloud, edge, laptop GPU) similar to our `cudaclaw` and `flux-runtime` concepts.

### 2. What's New/Hot (Specific Examples)
*   **LangGraph + LangSmith Evolution:** The focus has shifted from just building agentic workflows to **production observability, testing, and persistent memory**. The "StateGraph" paradigm is being extended with tools for debugging entire multi-agent sessions and exporting them as datasets—a parallel to the fleet's "tile" training system.
*   **CrewAI's "Enterprise" Trajectory:** Recent commits and discussions indicate a strong push towards **secure, auditable workflows** for corporate environments. Features being highlighted include granular permission controls, compliance logging, and integration with existing corporate auth systems. This suggests a market move from hobbyist prototyping to departmental deployment.
*   **AutoGen's Research Agent Specialization:** Microsoft's AutoGen is seeing increased activity around **specialized agents for code execution, scientific literature review, and data analysis**. The "User Proxy Agent" pattern is being refined for safer tool use, which aligns with the fleet's P0 (negative space) doctrine of defining what an agent should *not* do.
*   **Rise of the "Runtime" Layer:** Newer projects (e.g., **Braintrust**, **Agenta**) are less about the agent orchestration logic itself and more about the **deployment and evaluation platform**. They treat an agent workflow as a versioned artifact that can be A/B tested, monitored for cost/performance, and rolled back. This mirrors the fleet's separation of the `plato-torch` training room from the exported `plato-ensign` adapter.

### 3. Emerging Pattern: The Self-Improving Loop
A notable trend across several projects is the explicit design for **self-improvement via recorded interactions**. The pattern is: 1) Run agents, 2) Log inputs/outputs/traces, 3) Use those logs as fine-tuning data or prompt-optimization fuel, 4) Deploy the improved agent. This is a direct corollary to the fleet's core architecture: "Rooms train from accumulated interaction tiles."

### 4. Fleet Context & Differentiation
The Cocapn fleet's approach, as documented in `FLEET-CONTEXT.md`, has notable parallels and divergences:
*   **Parallel:** The concept of specialized roles (Navigator, Scout, etc.) is mainstream. The use of interaction data for training is a recognized frontier.
*   **Divergence:** The fleet's **Deadband Protocol (P0/P1/P2)** is a distinct formalization of constraint-first reasoning. The **"ghost tile"** concept (persisting lessons from dead agents) is a unique approach to institutional memory not commonly seen in open-source frameworks. The deep integration with a specific hardware stack (`Jetson`, `RTX 4050`) via `cudaclaw` is also a specialized, performance-oriented path.

### 5. Recommendation for Fleet Awareness
The fleet's `plato-ensign` (exporting room experience to LoRA) positions it well within the "self-improving loop" trend. To avoid "hitting rocks" (P0), the fleet should monitor how these open-source projects handle **data privacy** and **validation** of training logs, as our "zero-trust contributor" and server-side tile validation are critical for integrity.
