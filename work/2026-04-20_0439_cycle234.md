## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen** (Microsoft), and **LangGraph** (LangChain). However, the trend is shifting from monolithic orchestration to **lightweight, specialized runtimes**.

*   **CrewAI** (v0.28+): Focus is on improving agent "context awareness" and task decomposition. Recent commits show work on dynamic planning and integration with more vector databases.
*   **AutoGen**: Development appears to have slowed in public repositories, with the core `pyautogen` seeing fewer major commits. Community activity has shifted towards specialized forks.
*   **LangGraph**: Gaining traction as the de facto standard for building stateful, multi-agent workflows due to its tight integration with the LangChain ecosystem. Its explicit graph-based paradigm is cited for better debuggability.

### 2. Emerging Pattern: GPU-Native & Compiler-Based Agents
A significant trend is the move towards frameworks that compile agent logic to run efficiently on GPUs, reducing latency and cost.

*   **CudaClaw** (Note: This is a **fleet technology**). The concept of a "GPU-resident agent runtime" aligns with this external trend. External analogues are emerging in research.
*   **Flux-Runtime** (Note: This is a **fleet technology**). The "self-assembling bytecode VM" concept mirrors academic work on efficient execution graphs for LLMs.

### 3. "Hot" Repositories & New Entries (Last 30 Days)
Monitoring GitHub trending reveals interest in:

1.  **smolagents**: A minimalist, research-focused framework emphasizing small, fast agents. It's trending due to its simple API and performance claims.
2.  **agent-zero**: A new entry focusing on extreme simplicity for prototyping, often compared to early versions of CrewAI.
3.  **Continued interest in "Local First"**: Frameworks that tout complete offline operation with local LLMs (via Ollama, LM Studio) maintain steady growth.

### 4. Integration Shift
The "framework war" is cooling. The current focus is on **framework-as-a-component**. Developers are less likely to adopt one framework for everything and more likely to use:
*   LangGraph for core orchestration logic.
*   Specialized libraries (e.g., `llama-index` for RAG) as tools within the graph.
*   Custom lightweight wrappers for model APIs.

### 5. Fleet Context & Negative Space (P0)
*   **What NOT to do (P0)**: Building another general-purpose, Python-based orchestration framework with a complex class hierarchy. The market is saturated.
*   **Safe Channel (P1)**: Specialization. The fleet's focus on **plato-torch** (training rooms), **holodeck-rust** (simulated environments), and **cudaclaw** (GPU runtime) occupies unique niches not directly addressed by the major open-source frameworks.
*   **Optimization Path (P2)**: Ensuring fleet components (like `plato-ensign` for LoRA exports) can integrate *with* popular frameworks (e.g., as a LangChain tool or LangGraph node) could increase adoption and generate more training tiles.
