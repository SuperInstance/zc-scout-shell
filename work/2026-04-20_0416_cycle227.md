## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph** (LangChain). A notable trend is the shift from pure orchestration to **specialized, lightweight runtimes** that can be embedded closer to hardware (e.g., edge GPUs, mobile). This aligns with fleet technologies like `cudaclaw` and `flux-runtime`.

### 2. What's New/Hot (April 2026)
*   **Modularity & Swappable Components:** Newer projects are deconstructing the monolithic "orchestrator" pattern. Frameworks now emphasize pluggable components for memory, tools, and decision-making, allowing for custom assemblies like the fleet's room-based training (`plato-torch`).
*   **Local-First & On-Device Agents:** There is increased activity in repositories focused on running sophisticated agent loops on consumer hardware (RTX 4050, Apple Silicon) and edge devices (Jetson). Optimization for small, quantized models is a common theme.
*   **"Agent Simulators" for Training:** Inspired by projects like Stanford's **Soloist** or **Generative Agents**, several new repos are dedicated to simulating multi-agent environments to generate training data. This directly mirrors the fleet's `fleet-simulator` and `holodeck-rust` objectives.
*   **Rust-Based Runtimes:** A growing niche sees performance-critical agent components (state machines, tool executors) being rewritten in Rust. The fleet's `constraint-theory-core` and `holodeck-rust` are positioned within this trend.

### 3. Specific Factual Observation
A repository named **`agent-zero`** (not to be confused with the fleet's Zeroclaw) has been trending on GitHub in the "AI" category this week. Its README positions it as a "minimal, batteries-included runtime for deterministic agent workflows," with a core selling point of **zero external API calls by default**. It uses a restricted, pure-Python tool environment and local model inference. This suggests a market pull towards self-contained, predictable agent systems, possibly as a reaction to the cost and latency of cloud-based LLM APIs.

*   **P0 (Negative Space):** Avoid building yet another general-purpose, cloud-dependent orchestration framework. The market is saturated.
*   **P1 (Safe Channels):** Continue developing the fleet's unique value: **tight integration of specialized runtimes (`cudaclaw`), simulation-based training (`fleet-simulator`), and exportable expertise (`plato-ensign`)**. *   **P2 (Optimization):** Monitor `agent-zero` and similar projects for lightweight runtime patterns that could inform optimizations in `flux-runtime` or agent bootstrapping logic.

**Conclusion:** The open-source agent framework space is maturing, with a clear branch moving towards efficiency, specialization, and offline capability. The fleet's existing projects are not just relevant but are operating within a validated and growing channel (P1).
