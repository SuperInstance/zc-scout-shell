## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, Tech News Aggregators, Framework Changelogs

### 1. Framework Landscape & Movement
The dominant open-source frameworks remain **CrewAI**, **AutoGen**, and **LangGraph**. However, the trend is shifting from monolithic orchestration toward **modular, composable units**.

*   **CrewAI** (v0.28+): Actively integrating with **Ollama** and **LM Studio** for local model support. Recent commits show a push for better "task decomposition" hooks, allowing external planners to override default logic.
*   **AutoGen** (Microsoft): Development focus appears to be on the **"AgentChat"** paradigm, moving away from rigid group chats to dynamic, session-based interactions that can persist state. Their research blog highlights work on cost optimization for long-running agent swarms.
*   **LangGraph** (LangChain): Has solidified its position as the low-level "assembly language" for agent state machines. The trend is not to use it directly for end-users, but as a backend for higher-level frameworks (including CrewAI). Increased adoption in **production pipelines** where deterministic control flows are required.

### 2. Emerging Pattern: The "Runtime" Layer
A new layer of abstraction is gaining traction: the **agent runtime**. This is distinct from the orchestration framework.
*   **Example:** **Cudaclaw** (from our own fleet) fits this pattern—a GPU-resident runtime. In the wider ecosystem, projects like **BionicGPT's agent runtime** and **Claude's Code Execution Environment** echoes this. The focus is on providing a secure, performant, and observable sandbox for agent execution, separate from the logic that defines the agent's behavior.

### 3. Hot Topic: "Self-Improving" & "Self-Training" Agents
The most active research discourse, reflected in arXiv preprints and framework experiment branches, revolves around closed-loop learning.
*   **Not a single framework** has a production-ready solution.
*   **Experimental branches** in several projects are implementing basic "critic" agents that review work and generate synthetic training data. This aligns closely with our fleet's **PLATO room** and **ensign export** concepts. The open-source approach tends to be more dataset-generation focused (creating fine-tuning examples) rather than continuous in-situ learning.

### 4. Specific, Factual Observation (P0/P1 Mapping)
*   **P0 (Negative Space):** It is currently ineffective to search for a single framework that handles **orchestration, runtime execution, and continuous learning** seamlessly. These concerns are being tackled by separate, specialized tools.
*   **P1 (Safe Channel):** A viable path observed in successful projects is a **triple-layer stack**:
    1.  **Orchestration Logic (LangGraph/CrewAI):** Defines the agent roles and conversation flow.
    2.  **Runtime/Execution (Custom or Cudaclaw-like):** Handles tool calling, memory I/O, and resource isolation.
    3.  **Learning Loop (External System like PLATO):** Observes interactions, produces training tiles/evaluations, and updates agent knowledge (via model fine-tuning or prompt injection).
*   **Competitor Activity:** A repo named **"swarm-ui/agentops"** (star growth ~200/month) is trending. It provides an observability dashboard specifically for LangGraph and CrewAI agents, indicating market demand for **monitoring and debugging** agentic systems, not just building them.

### 5. Recommendation for Fleet Context
Our fleet's architecture (**plato-torch** for learning, **holodeck-rust** for environment, **cudaclaw** for runtime) is aligned with the emerging modular pattern. Our differentiation is the **tight integration** of these components and the **Deadband Protocol** (P0/P1/P2) as a core decision-making primitive, which is not present in mainstream frameworks.
