**Focus:** Frameworks for building, orchestrating, and deploying AI agents.

### 1. **Established Leaders (Maturing)**
*   **CrewAI:** Strong focus on role-based agent collaboration. Recent activity shows increased emphasis on production deployment tools and integration with external data sources (vector databases, APIs).
*   **LangGraph** (LangChain): The dominant paradigm for defining agents as stateful graphs. Most new research prototypes and commercial products reference LangGraph as their underlying orchestration layer. The `StateGraph` abstraction is becoming a de facto standard.
*   **AutoGen** (Microsoft): Maintains a strong presence in academic and research contexts, particularly for multi-agent conversations and code generation. Less visible in trending for pure deployment, but a key library for agent simulation.

### 2. **Emerging & Trending Patterns**
*   **Specialization over Generalization:** Newer frameworks are not trying to be "everything." They are focusing on specific niches:
    *   **Web/UI Agents:** Frameworks like `open-webui` and specialized libraries for browser automation (Playwright, Selenium wrappers) are gaining stars. The focus is on reliable, observable interaction with GUIs.
    *   **Data & Analytics Agents:** Tight integration with pandas, SQL databases, and BI tools. Agents that can generate and explain plots, write data transformation code.
    *   **DevOps/Infrastructure Agents:** Frameworks that can parse Terraform, Kubernetes manifests, and cloud SDKs to manage infrastructure.
*   **The "Runtime" Shift:** Discussion is moving from pure Python libraries to dedicated, containerized **agent runtimes**. Concepts like long-running processes, health checks, telemetry, and hot-swapping of agent logic (like our **cudaclaw** and **flux-runtime**) are in early research phases. The goal is to move agents from scripts to managed services.
*   **Cost & Latency as Primary Constraints:** Architectural discussions heavily prioritize token budgeting, caching strategies, and model routing (e.g., using a small, fast model for classification, a large model for reasoning). This aligns with fleet doctrine's P0/P1 thinking—avoiding cost overruns is a primary negative space.
