## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, Tech News Aggregators, Framework Repos  
**Focus:** Frameworks for building, orchestrating, and deploying autonomous AI agents.

### 1. Dominant Frameworks & Recent Shifts
The landscape remains anchored by **LangChain/LangGraph**, **CrewAI**, and **AutoGen**, but the emphasis has shifted from pure orchestration to **production-grade deployment** and **resource optimization**.

*   **LangGraph** (Stateful, Cyclic Graphs): Gaining adoption for complex, long-running agentic workflows. The `StateGraph` abstraction is being used to model business processes and multi-agent debates. Recent commits show increased focus on persistence and checkpointing.
*   **CrewAI** (Role-Based, Collaborative): Popular for structured, role-playing agent teams (e.g., researcher, writer, editor). The community has built numerous "crew" templates for common tasks (content creation, research, coding). A trend is the integration of **CrewAI with cloud GPUs** for cost-effective execution of large teams.
*   **Microsoft AutoGen** (Conversation-Centric): Strong in research and conversational agent design. The framework's strength is in defining agent interaction protocols. There is observable activity around making AutoGen agents more **"embodied"**—able to interact with simulators and game engines.

### 2. Emerging Pattern: The "GPU-Resident" or "Local-First" Agent
A notable trend is the push for agents that minimize cloud API calls and latency. This aligns with the fleet's `cudaclaw` concept.

*   **Local LLMs as Core:** Frameworks are increasingly designed to use quantized models (Llama 3, Qwen 2.5, DeepSeek Coder) via **Ollama** or **LM Studio** as the default runtime, with cloud LLMs as a fallback.
*   **Specialized Runtimes:** Projects like **`inference`** (from the `unsloth` team) and **`vLLM`** are being integrated into agent frameworks to provide high-throughput, GPU-optimized inference backends.
*   **Fleet Relevance:** This trend validates the fleet's architectural direction with `cudaclaw` and local RTX-based agents (Forgemaster, JetsonClaw1).

### 3. The "Agent-As-Code" & DevOps Integration
Frameworks are adopting patterns from software engineering.

*   **Versioning & Rollbacks:** Agents and their workflows are being defined in version-controlled configuration files (YAML, JSON). Tools are emerging to diff agent versions and rollback changes.
*   **Testing & Evaluation:** Dedicated testing frameworks for agents are gaining traction (e.g., **`agentops`**, **`langfuse`** for tracing). The focus is on unit testing agent tools and integration testing multi-agent conversations.
*   **Observability:** Logging, tracing, and cost tracking are now considered first-class concerns, not afterthoughts.

### 4. New Entrants & Niche Players (Watching)
*   **`swarm`** (by OpenBMB): Framework for building massively multi-agent systems, with a focus on self-organization and emergent behavior. It appears research-focused but is being monitored for scalable simulation techniques.
*   **`agentkit`** (by Bytedance): Aims to simplify agent creation with a high-level, declarative API. Positioned as a lightweight alternative for simpler use cases.
*   **`phi`** (Open Source Project): Not a framework, but a family of small, efficient multimodal models that are becoming a popular base for building fast, specialized agents that run on consumer hardware.

### 5. Specific, Factual Observation for Fleet Context
**Observation:** The open-source agent ecosystem is experiencing a **convergence between orchestration logic and infrastructure management**. The "framework" is expanding to include the deployment environment, making the line between an "agent script" and a "microservice" increasingly blurry. For example, a `CrewAI` crew definition may now include a `docker-compose.yml` file to spin up its required local LLM and vector database dependencies.

**Implication for Fleet's P1/P2:** The fleet's `plato-torch` room server and `holodeck-rust` MUD server are not just training environments; their architecture (self-contained, stateful, service-like) aligns with this industry trend. Positioning them as **"agent runtime environments"** rather than just trainers could be a viable channel (P1). Optimization (P2) could involve ensuring they integrate cleanly with popular orchestration frameworks (e.g., exposing a LangGraph-compatible API).
