## Output: Scout Report — Open-Source Agent Frameworks (April 2026)

**Observation Date:** 2026-04-20  
**Source:** GitHub Trending, AI/ML community forums, project releases.

### 1. Dominant Frameworks & Shifts
The landscape remains anchored by **CrewAI**, **AutoGen**, and **LangGraph**, but a significant shift toward **lightweight, single-binary runtimes** is observable. Projects like **Cline** (written in Zig) and **MCP** (Model Context Protocol) clients are gaining traction for their minimal dependencies and fast startup times. The trend suggests a move away from heavy Python orchestrators for certain embedded or CLI-centric agent tasks.

### 2. Emerging Pattern: "GPU-Resident" Agents
Following the release of NVIDIA's **NIMs** and the **TensorRT-LLM** runtime, several projects now advertise "GPU-resident" or "zero-copy" agent loops. This aligns with fleet technology **cudaclaw**. A notable example is **llama.cpp**'s `server` mode with continuous batching, which allows multiple agent instances to share a single loaded model without CPU-GPU transfer penalties. This pattern is particularly relevant for real-time simulation environments like **fleet-simulator**.

### 3. Integration with Open-Source Models
Frameworks are rapidly adding native support for **DeepSeek-V3**, **Qwen2.5**, and **Llama 3.1** model families. The **Ollama** integration has become a de facto standard for local testing; most frameworks now include an `OllamaProvider` class. There is increased experimentation with **mixture-of-experts (MoE)** models for agent routing, though production deployments appear limited.

### 4. Specific Hot Repo (Last 7 Days): **smolagents**
A repository named **smolagents** (by a group called `huggingface`) has appeared on trending. It promotes a "minimal, predictable" API for constructing agent loops, with built-in support for Hugging Face tools and models. Its differentiation is a strict focus on reducing non-determinism in tool calling—a direct response to criticism of stochastic planning in earlier frameworks. This aligns with the fleet's **Deadband Protocol** emphasis on finding safe, predictable channels (P1) before optimization.

### 5. Cautionary Note (P0: Negative Space)
Several projects attempting to implement "fully autonomous" agent swarms with circular tool-calling dependencies have shown high failure rates in reproducibility tests. Community reports indicate issues with infinite loops and credit exhaustion on paid API endpoints. The safe channel (P1) currently appears to be **single-agent with deterministic tool planning** or **small, tightly coordinated crews with finite state machines**.

### 6. Fleet Relevance
- **plato-torch** rooms could integrate **smolagents**-style deterministic tool planners to generate more reliable training tiles.
- The GPU-resident pattern validates the fleet's investment in **cudaclaw** as a competitive architecture.
- The trend toward lightweight runtimes suggests **flux-runtime** (self-assembling bytecode VM) may find a niche if it can demonstrate lower latency than Zig/Rust alternatives.

**Recommendation:** Monitor **smolagents** for API stability. Consider a test integration to see if its deterministic tool engine reduces noisy tiles in training rooms.
